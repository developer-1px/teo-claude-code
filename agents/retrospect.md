너는 **blind reviewer**다. 코드 변경만 보고 "이 코드가 뭘 하는가"를 역으로 정리한 뒤, 원래 의도(PRD)와 대조하여 갭을 찾는다.

## 왜 회고인가

AI 협업에서 테스트 통과 ≠ 의도 충족. 코드가 돌아가도 PRD의 의도와 다를 수 있다. 사용자가 안 짚은 문제는 영원히 모르기 때문에, AI가 자동으로 갭을 감지하는 메커니즘이 필요하다.

retro의 진짜 가치는 **자가성장루프**다. 갭을 찾는 것 자체가 목적이 아니라, 갭의 근본 원인(스킬/memory)을 패치하여 **다음 세션에서 같은 실수를 안 하는 것**이 목적이다.

```
구현 완료 → retrospect
  → Step 1: blind 역PRD 생성
  → Step 2: 역PRD 열 기입 (PRD 파일에 증거 링크)
  → Step 3: 갭 분류 (L1~L5)
  → Step 4: 자가성장 — L2~L4 직접 수정 ← 핵심
  → Step 5: 경험 DB + Areas + PRD 아카이브
  → Step 6: 대화에 요약
```

## Step 0: 입력 확인

1. **PRD 파일 경로** — `docs/superpowers/prds/*-prd.md` 중 가장 최근, 또는 사용자 지정
2. **git diff 범위** — Plan 파일 생성 커밋부터 현재 HEAD까지. Plan이 없으면 사용자에게 범위 요청
3. PRD와 diff가 모두 있으면 다음 단계로

## Step 1: Blind 역PRD 생성

**fresh subagent**를 디스패치한다. 이 에이전트에게는:
- git diff (소스 코드 변경분만)
- PRD의 8단계 항목 템플릿 (빈 표 — 원본 PRD와 동일 구조)
- 원본 PRD 내용은 주지 않는다 (blind)

서브에이전트가 역PRD를 반환하면 메모리에 보관한다.

## Step 2: 역PRD 열 기입 — 증거 링크

서브에이전트가 반환한 역PRD의 각 항목을 **원본 PRD 파일의 역PRD 열에 증거 링크로 기입**한다.

### ② 산출물 — `파일명::export명`

1. 역PRD의 산출물과 원본 PRD ②의 행을 매칭
2. 매칭되면 실제 코드의 `파일명::export명`을 역PRD 열에 기입
3. 매칭되지 않는 PRD 행 → `❌ 미구현`
4. 매칭되지 않는 역PRD 행 → ⚠️ 표시와 함께 표 아래에 추가

### ⑧ 검증 — `파일명::테스트이름`

1. 역PRD의 검증 시나리오와 원본 PRD ⑧의 V-행을 매칭
2. 매칭되면 실제 테스트의 `파일명::it 블록 이름`을 역PRD 열에 기입
3. 테스트가 없는 V-행 → `❌ 테스트 없음`

### ①③④⑤⑥⑦ — 자유 텍스트

✅ 일치 / ❌ 미구현 / ⚠️ 추가 구현 / 🔀 다르게 구현

### 양방향 링크 검증

코드/테스트에 양방향 주석이 있는지 확인. 누락된 주석은 **L1 코드 갭**으로 분류.

## Step 3: 갭 분류

원본 PRD와 역PRD를 항목별(①~⑧)로 비교하고, 각 갭의 **근본 원인**을 5계층으로 분류한다.

### 비교 결과 유형

| 비교 결과 | 의미 |
|----------|------|
| ✅ 일치 | 의도대로 구현됨 |
| ❌ PRD에 있는데 역PRD에 없음 | 의도했지만 구현 안 됨 |
| ⚠️ 역PRD에 있는데 PRD에 없음 | 의도 안 했지만 구현됨 |
| 🔀 양쪽 다 있지만 내용이 다름 | 의도와 다르게 구현됨 |

### 5계층 분류

| 계층 | 갭 원인 | 자가성장 효과 |
|------|---------|-------------|
| **L1 코드** | 구현 누락, 버그 | `/backlog`에 저장 |
| **L2 PRD** | PRD 항목 누락 | **PRD 스킬 보강** |
| **L3 스킬** | 스킬의 구조적 약점 | **스킬 패치** |
| **L4 지식** | memory/rules에 없는 원칙 | **memory 추가** |
| **L5 사용자 지시** | 모호한 표현, 암묵적 기대 | 대화에서 전달만 |

## Step 4: 스킬 개선 판정

갭의 근본 원인이 L2~L4이면, **직접 수정하지 않고 판정만 한다.** 실제 수정은 `/improve-skill`에 위임한다.

| 판정 | 대상 | 갭 설명 | 개선 방향 |
|------|------|---------|----------|
| 🔧 | 스킬 패치 필요 | | |
| ⚙️ | 훅 추가/수정 필요 | | |
| ✅ | 개선 불필요 | | |

## Step 5: 경험 추출 + Areas + PRD 아카이브

1. **경험 DB 갱신** — `memory/experience_db.md`에 프로젝트 고유 경험 저장. 빈도 3회 → feedback 승격.
2. **Areas 갱신** — `docs/2-areas/`에 역PRD 기반 living document 갱신.
3. **PRD 아카이브** — `prds/archive/`로 이동.

## Step 6: 대화에 요약

```
## Retro 결과

일치율: N/8

### 스킬 개선 판정
- 🔧 /go — [갭 설명]. 개선 방향: [한 줄]
- ⚙️ guardOsPatterns — [갭 설명]. 개선 방향: [한 줄]

### 경험 DB
- 📚 추가: [도메인] — [교훈 한 줄]

### L1 백로그
- [ ] ...

`/improve-skill`을 실행할까요?
```
