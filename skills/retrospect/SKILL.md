## 역할

너는 **blind reviewer**다. 코드 변경만 보고 "이 코드가 뭘 하는가"를 역으로 정리한 뒤, 원래 의도(PRD)와 대조하여 갭을 찾는다.

## 왜 회고인가

AI 협업에서 테스트 통과 ≠ 의도 충족. 코드가 돌아가도 PRD의 의도와 다를 수 있다. 사용자가 안 짚은 문제는 영원히 모르기 때문에, AI가 자동으로 갭을 감지하는 메커니즘이 필요하다.

retro의 진짜 가치는 **자가성장루프**다. 갭을 찾는 것 자체가 목적이 아니라, 갭의 근본 원인(스킬/memory)을 패치하여 **다음 세션에서 같은 실수를 안 하는 것**이 목적이다.

```
구현 완료 → /retrospect
  → Step 1: blind 역PRD 생성
  → Step 2: 역PRD 열 기입 (PRD 파일에 증거 링크)
  → Step 3: 갭 분류 (L1~L5)
  → Step 4: 자가성장 — L2~L4 직접 수정 ← 핵심
  → Step 5: 경험 DB + Areas + PRD 아카이브
  → Step 6: 대화에 요약
```

## Step 0: 입력 확인

1. **PRD 파일 경로** — `docs/superpowers/prds/*-prd.md` 중 가장 최근, 또는 사용자 지정
2. **git diff 범위** — Plan 파일 생성 커밋부터 현재 HEAD까지. Plan이 없으면 사용자에게 범위 요청
3. PRD와 diff가 모두 있으면 다음 단계로

## Step 1: Blind 역PRD 생성

**fresh subagent**를 디스패치한다. 이 에이전트에게는:
- ✅ git diff (소스 코드 변경분만)
- ✅ PRD의 8단계 항목 템플릿 (빈 표 — 원본 PRD와 동일 구조)
- ❌ 원본 PRD 내용은 주지 않는다 (blind)

서브에이전트 프롬프트:

```

아래 8단계 항목의 빈 표를 채우세요. 코드에서 파악할 수 있는 것만 채우고, 파악할 수 없는 것은 "코드에서 파악 불가"로 표기하세요.
```

## Step 2: 역PRD 열 기입 — 증거 링크

서브에이전트가 반환한 역PRD의 각 항목을 **원본 PRD 파일의 역PRD 열에 증거 링크로 기입**한다. ✅/❌ 자기 선언이 아니라 **추적 가능한 증거**를 남긴다.

### ② 산출물 — `파일명::export명`

1. 역PRD의 산출물과 원본 PRD ②의 행을 매칭한다
2. 매칭되면 실제 코드의 `파일명::export명`을 역PRD 열에 기입한다
3. 코드가 합성(middleware/config)으로 여러 행을 압축하면 N:1로 같은 export를 가리켜도 된다
4. 매칭되지 않는 PRD 행 → `❌ 미구현`
5. 매칭되지 않는 역PRD 행(PRD에 없는 것이 구현됨) → ⚠️ 표시와 함께 표 아래에 추가

```markdown
| 산출물 | 설명 | 역PRD |
|--------|------|-------|
| `tab.ts` | tab 축, 4개 전략 | ✅ `tab.ts::tab` |
| `spatial.ts` 수정 | ... | ❌ 미구현 |
```

### ⑧ 검증 — `파일명::테스트이름`

1. 역PRD의 검증 시나리오와 원본 PRD ⑧의 V-행을 매칭한다
2. 매칭되면 실제 테스트의 `파일명::it 블록 이름`을 역PRD 열에 기입한다
3. 테스트가 없는 V-행 → `❌ 테스트 없음`
4. 테스트는 있지만 `// V{n}:` 주석이 없으면 → 증거 링크 기입 후, 주석 누락을 **L1 코드 갭**으로 분류

```markdown
| # | 시나리오 | 예상 결과 | 역PRD |
|---|---------|----------|-------|
| V1 | Tab → 다음 노드 | DFS 이동 | ✅ `cms-tab-flow.integration.test.tsx::Tab moves focus in DFS order` |
| V3 | 마지막 노드 Tab | zone 탈출 | ❌ 테스트 없음 |
```

### ①③④⑤⑥⑦ — 기존과 동일

자유 텍스트로 매칭 결과를 기입한다 (✅ 일치 / ❌ 미구현 / ⚠️ 추가 구현 / 🔀 다르게 구현).

### 양방향 링크 검증

역PRD 열을 기입한 후, 코드/테스트에 양방향 주석이 있는지 확인한다:
- 코드: `// ② PRD파일명` 주석이 해당 export에 있는지
- 테스트: `// V{n}: PRD파일명` 주석이 해당 it 블록에 있는지
- 누락된 주석은 **L1 코드 갭**으로 분류

## Step 3: 갭 분류

원본 PRD와 역PRD를 항목별(①~⑧)로 비교하고, 각 갭의 **근본 원인**을 5계층으로 분류한다.

### 비교 결과 유형

| 비교 결과 | 의미 |
|----------|------|
| ✅ 일치 | 의도대로 구현됨 |
| ❌ PRD에 있는데 역PRD에 없음 | 의도했지만 구현 안 됨 |
| ⚠️ 역PRD에 있는데 PRD에 없음 | 의도 안 했지만 구현됨 (PRD가 덜 촘촘했다는 신호) |
| 🔀 양쪽 다 있지만 내용이 다름 | 의도와 다르게 구현됨 |

### 5계층 분류 — 근본 원인 추적

**갭을 찾으면 "왜?"를 반복하여 근본 원인 계층까지 도달한다.** 코드 갭(L1)에서 멈추지 않고, 그 코드 갭을 만든 상위 원인을 추적하는 것이 핵심이다.

| 계층 | 갭 원인 | 자가성장 효과 |
|------|---------|-------------|
| **L1 코드** | 구현 누락, 버그, 잘못된 로직 | `/backlog`에 저장 |
| **L2 PRD** | PRD 항목이 비어있었음, 체크리스트 누락 | **PRD 스킬 보강 → 다음 PRD에서 누락 방지** |
| **L3 스킬** | Discussion/Plan/Go 스킬의 구조적 약점 | **스킬 패치 → 다음 세션에서 더 나은 워크플로** |
| **L4 지식** | memory/rules에 없는 원칙 | **memory 추가 → 다음 세션에서 자동 적용** |
| **L5 사용자 지시** | 모호한 표현, 암묵적 기대 | 대화에서 사용자에게 전달 |

**분류 기준 — "왜?" 체인:**
- "코드에서 뭘 안 했는가?" → L1
- "PRD에 왜 안 적혀 있었는가?" → L2
- "PRD에 적혀야 하는 걸 스킬이 왜 안 물어봤는가?" → L3
- "이런 원칙이 있었다면 잡혔을 텐데?" → L4
- "사용자가 이걸 명시했다면 잡혔을 텐데?" → L5

**L2~L4가 자가성장루프의 핵심이다.** L1은 코드 수정일 뿐이고, L5는 사용자 영역이다. L2~L4를 찾아서 패치하는 것이 retro의 진짜 산출물.

## Step 4: 스킬 개선 판정

**retro의 핵심 단계.** 갭의 근본 원인이 L2~L4이면, **직접 수정하지 않고 판정만 한다.** 실제 수정은 `/improve-skill`에 위임한다.

### 판정 기준

각 갭에 대해:
1. **"이 갭을 다음 세션에서 방지하려면 뭘 바꿔야 하는가?"**
2. 답이 스킬/훅이면 → 🔧 개선 필요
3. 답이 "이번만 특수한 상황"이면 → ✅ 개선 불필요

### 판정 산출물

| 판정 | 대상 | 갭 설명 | 개선 방향 (한 줄) |
|------|------|---------|-----------------|
| 🔧 | /go | 훅 파일 작성 후 통합 검증 안 함 | verify 단계에 훅 테스트 추가 |
| ⚙️ | guardOsPatterns | useState(isOpen) 미감지 | regex 패턴 확장 |
| ✅ | — | 사용자가 PRD 생략 선택 | 개선 불필요 |

- 🔧 = 스킬 패치 필요
- ⚙️ = 훅 추가/수정 필요
- ✅ = 개선 불필요

L1은 `/backlog`에 저장하거나 즉시 수정.
L5는 대화에서 사용자에게 전달만.

## Step 5: 경험 추출 + Areas + PRD 아카이브

### 경험 DB 갱신

이번 retro에서 **프로젝트 고유 경험**을 추출하여 `memory/experience_db.md`에 저장한다.

**프로젝트 고유 판정 (리트머스):** 교훈에 이 프로젝트의 파일명, 타입명, 아키텍처 패턴이 들어가면 프로젝트 고유 → 저장. 일반 코딩 지식은 LLM이 이미 알고 있으므로 저장하지 않는다.

**패턴 매칭:** 같은 교훈이 있으면 빈도 +1, 없으면 새 행 추가.

**빈도 3회 → feedback 승격:** 반복되는 교훈은 `type: feedback` memory 파일로 승격하여 매 세션에서 자동으로 읽히게 한다.

### Areas 갱신

역PRD 기반으로 `docs/2-areas/{도메인}.md`를 갱신한다. Areas는 "이 도메인이 지금 어떻게 동작하는가"를 담는 도메인축 living document.

### PRD 아카이브

retro 완료된 PRD를 `prds/archive/`로 이동. `prds/`에는 진행 중인 PRD만 남긴다.

```bash
git mv docs/superpowers/prds/YYYY-MM-DD-feature-prd.md docs/superpowers/prds/archive/
```

## Step 6: 대화에 요약

모든 작업이 끝나면 대화에 핵심만 보여준다:

```
## Retro 결과

일치율: N/8

### 스킬 개선 판정
- 🔧 /go — [갭 설명]. 개선 방향: [한 줄]
- ⚙️ guardOsPatterns — [갭 설명]. 개선 방향: [한 줄]
- ✅ 개선 불필요 — [이유]

### 경험 DB
- 📚 추가: [도메인] — [교훈 한 줄]
- 🎓 feedback 승격: [도메인] — [교훈] (3회 반복)

### L1 백로그
- [ ] ...

`/improve-skill`을 실행할까요?
```

🔧 또는 ⚙️ 항목이 하나라도 있으면 `/improve-skill` 실행을 제안한다.

## 종료 조건

- 역PRD 열 기입 완료 (PRD 파일에 증거 링크)
- **스킬 개선 판정 완료** ← 핵심 산출물
- L1 항목 → `/backlog`에 저장 또는 즉시 수정
- 경험 DB 업데이트 완료
- Areas 문서 갱신 완료
- PRD 아카이브 완료
- 대화에 요약 제시
- 🔧/⚙️ 판정이 있으면 → `/improve-skill` 실행 제안

⛔ **금지:**
- L5(사용자 피드백)는 자동 반영하지 않는다 — 대화에서 전달만 한다.
- retro에서 스킬/훅을 **직접 수정하지 않는다** — 판정만 하고 `/improve-skill`에 위임한다.
