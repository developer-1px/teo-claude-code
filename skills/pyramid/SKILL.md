## /pyramid — SCQAPD 피라미드 빌더

> **산출물**: `docs/0-inbox/pyramid-{slug}.md` — Mermaid graph TD + 노드 목록
> **원칙**: 사용자가 던지는 모든 말은 피라미드의 재료다. 분류하고, 연결하고, 빈칸을 질문한다.

---

### SCQAPD 노드 타입

| 타입 | 역할 | 색상 |
|------|------|------|
| **S** Situation | 현재 상태, 배경 사실 | `fill:#e3f2fd,stroke:#1565c0` |
| **C** Complication | 긴장, 문제, 변화 | `fill:#fce4ec,stroke:#c62828` |
| **Q** Question | S+C에서 떠오르는 핵심 질문 | `fill:#fff3e0,stroke:#e65100` |
| **A** Answer | Q에 대한 답 = 핵심 주장 | `fill:#e8f5e9,stroke:#2e7d32` |
| **P** Proof | A를 뒷받침하는 근거 | `fill:#f3e5f5,stroke:#6a1b9a` |
| **D** Detail | P의 세부 사항, 데이터 | `fill:#eceff1,stroke:#455a64` |

---

### 핵심 루프 — 매 턴 수행

#### 1. 분류

사용자의 입력을 읽고 SCQAPD 중 어디에 해당하는지 판단한다.

- 하나의 입력이 여러 노드를 포함할 수 있다
- 판단이 애매하면 "이건 S(상황)으로 봤는데, C(전개)에 더 가까울까요?" 식으로 확인

#### 2. 연결

기존 노드와 연결할 수 있는지 판단한다.

**연결 규칙 — 엣지에는 질의 라벨을 붙인다:**

| 연결 | 기본 질의 | 예시 |
|------|----------|------|
| S → C | "그런데?" | "그런데 AI가 만든 UI는..." |
| C → Q | "그래서?" | "그래서 왜 중요한가?" |
| Q → A | "답은?" | "답은 비용효율 인프라" |
| A → P | "왜?" | "왜냐하면 매번 짜는 건..." |
| A → A | "구체적으로?" | "구체적으로 데이터 레이어는..." |
| P → D | "어떻게?" | "어떻게 보면 플러그인이..." |

- 기본 질의로 충분하면 그대로 사용
- 맥락이 있으면 구체적 질의로 대체 (예: `"왜?"` → `"왜 기존 라이브러리론 안 돼?"`)
- Mermaid 표현: `A1 -->|"왜?"| P1`

연결할 노드가 있으면 → 연결하고 그래프 갱신
연결할 노드가 없으면 → 갭 질문 (아래 3번)

#### 3. 갭 질문 — 균형 우선 전략

매 턴 파일의 현재 피라미드를 읽고, **가장 빈약한 층**을 찾아 질문한다. 위에서 아래로 순서대로 드릴다운하지 않는다.

**균형 판정:**
1. 피라미드 현황(S/C/Q/A/P/D 카운트)을 확인한다
2. 도입부(S+C)가 충분하면(각 1개 이상) → 아래층(A/P/D)을 우선 채운다
3. 도입부가 부족하면 → 도입부를 먼저 채운다
4. 특정 층이 비대하면 → 압축을 제안한다 (예: S 5개 → 2개로)

**빈약한 층 판정 기준:**
- A가 있는데 P가 부족 (P < A×2) → P를 채운다
- P가 있는데 D가 없음 → D를 채운다 (구체적 예시/데이터)
- S+C가 A+P+D보다 많으면 → 도입부 비대, 아래층 채우기 우선

**갭 질문 유형:**
1. **아래로 확장**: "왜 그렇게 생각하나요? 근거는?" (P 부족 시)
2. **옆으로 확장**: "다른 근거가 더 있나요?" (MECE 점검)
3. **구체화**: "구체적 예시나 수치가 있나요?" (D 부족 시)
4. **도입부 보강**: "이 상황에서 뭐가 문제인가요?" (S/C 부족 시)
5. **압축 제안**: "S가 5개인데 2개로 합칠 수 있을까요?" (비대 시)

#### 수평 관계 판정 — 연역 vs 귀납

같은 부모 아래 P/D가 2개 이상 모이면, 수평 관계를 판정하고 그에 맞게 질문한다.

**연역 (전제→전제→결론 체인):**
- 신호: P들이 순서 의존적. "P1이 참이면 P2가 따라오고, P2가 참이면 결론"
- 질문: "이 전제 체인에 빠진 고리가 있나요?" / "P1이 거짓이면 나머지도 무너지나요?"
- Mermaid: `P1 -->|"그러면"| P2 -->|"따라서"| P3`

**귀납 (독립 관찰 → "그래서?" 결론):**
- 신호: P들이 독립적. 하나를 빼도 나머지가 성립
- 질문: "이 관찰들을 관통하는 패턴은?" / "빠진 관찰은 없나요?" (MECE)
- Mermaid: `A1 -->|"왜?"| P1` + `A1 -->|"왜?"| P2` + `A1 -->|"왜?"| P3`

**판정 후 행동:**
- 연역인데 고리가 빠져있으면 → "이 사이에 전제가 하나 더 필요한 것 같은데요"
- 귀납인데 MECE가 안 되면 → "이 축 외에 빠진 관점이 있을까요?"
- 판정이 애매하면 → "이 근거들은 순서가 있나요(연역), 독립인가요(귀납)?"

**질문 형식:** 크리티컬 프렌드 톤으로. 호기심 기반, 공격적 프레이밍 금지.
```
제 판단: [AI가 추론한 후보]. 이유: [근거].
한 가지 궁금한 게 있는데요 — [갭 질문]
```

### 레드팀 — 크리티컬 프렌드

피라미드를 채워가면서 **소크라테스식 질문**으로 사용자의 사고를 자극한다.

**톤 규칙:**
- "반론부터 던지겠습니다" ❌ → "한 가지 궁금한 게 있는데요" ✅
- "~아닌가?" (공격) ❌ → "~라고 볼 수도 있을까요?" (호기심) ✅
- 판단을 먼저 밝히되, 대화 상대가 편하게 반박할 수 있는 여지를 남긴다

**질문 유형:**
- **경계 질문**: "이게 ~에도 적용되나요, 아니면 ~까지만?"
- **반례 질문**: "혹시 ~인 경우엔 어떨까요?"
- **차별화 질문**: "기존의 ~와 비교하면 결정적 차이가 뭘까요?"
- **깊이 질문**: "그 이유를 한 단계 더 파면?"

#### 4. 산문 검증 (매 턴, 에이전트 위임)

매 턴 파일 갱신 후, **별도 에이전트(haiku)**에 산문 검증을 위임한다. 메인 대화 흐름을 막지 않는다.

**에이전트 프롬프트:**
```
피라미드 파일을 읽고, 모든 엣지를 산문으로 검증하라.
각 엣지: "[부모 노드 내용] — [엣지 질의] — [자식 노드 내용]"
→ 한 문장으로 자연스럽게 읽히면 ✅, 안 읽히면 ❌ + 이유.
파일의 "## 산문 검증" 섹션을 결과로 갱신하라.
```

**에이전트 설정:**
- model: haiku (비용 최소화)
- run_in_background: true
- 결과: 파일의 `## 산문 검증` 섹션을 직접 갱신

**❌ 발견 시 메인 에이전트가 사용자에게 보고:**
- "이 연결이 글로 안 읽히는데요 — [구체적 문장]. 재배치할까요?"

### 정렬 — 민토의 순서 원칙

같은 부모 아래 형제 노드가 **3~4개 이상** 모이면 자동으로 정렬을 점검한다. 순서가 맞아야 글로 읽힌다.

**트리거 조건:**
- 같은 부모 아래 형제(P, D 등)가 3개 이상 모인 시점
- 새 형제가 추가될 때마다 기존 순서가 여전히 맞는지 재점검
- 정렬은 강제가 아님 — 제안하고 사용자 확인 후 적용

**3가지 정렬 원칙 (하나만 선택):**

| 순서 | 기준 | 언제 사용 | 예시 |
|------|------|----------|------|
| **시간** | 인과/시간순 | "이것 때문에 저것이 생겼다" | S→C→Q (인과 체인) |
| **구조** | 그룹-항목 | "큰 덩어리 안에 작은 것들" | 5층 시스템(엔진→부품→배치→하네스→devtools) |
| **정도** | 유사성/중요도 | "비슷한 것끼리, 강한 것 먼저" | P1~P5를 임팩트 순으로 |

**정렬 절차:**

1. 같은 부모 아래 형제 노드 그룹을 식별한다 (예: A1 아래 P1~P5)
2. 현재 순서가 위 3가지 중 어떤 원칙을 따르는지 판정한다
3. 원칙이 없거나 섞여있으면 → 가장 적합한 원칙을 선택하고 재정렬 제안
4. 재정렬 시 Mermaid의 노드 선언 순서와 노드 목록 테이블 순서를 함께 바꾼다

#### 폭 제어 — 중간 그룹핑

같은 부모 아래 형제가 **5개 이상**이면 팬아웃이 과도하다. 2~3개 그룹으로 묶어 중간 레이어를 만든다.

**규칙:**
- 형제 5개 이상 → 의미 기준으로 2~3 그룹핑 제안
- 그룹 라벨은 subgraph 제목으로 표현 (번호 접두사: ①②③)
- **그룹은 병렬이 아니라 체인으로 연결** — 미괄식(약→강→결론)이 기본
- Mermaid: `A1 → G1 → G2 → G3` (팬아웃 1)

**미괄식 체인 패턴:**
```
A1 →|"먼저"| G1(약한 논거)
G1 →|"그래서"| G2(강한 논거)
G2 →|"결국"| G3(결론/기회)
```
폭 5 → 1(체인) × 2(그룹 내부). 읽는 순서가 곧 설득 순서.

**정렬 제안 형식:**
```
현재 P1~P5 순서: 비용→범위→형태→품질→시장
제 판단: "정도" 순서가 맞을 것 같습니다 — 임팩트가 큰 순서로.
제안: 품질→형태→범위→비용→시장 (품질이 가장 결정적, 시장은 파생)
이 순서가 맞나요?
```

**파일에 기록:**
노드 목록 테이블에 `순서` 컬럼을 추가하거나, 정렬 근거를 `## 정렬` 섹션에 기록한다.

```markdown
## 정렬

- A1 → P1~P5: **정도** 순서 (임팩트 순). 품질(P4)이 가장 결정적 — "없으면 위험"이 "있으면 좋다"보다 강하다.
```

### 완성 판정

피라미드가 완성되려면:
- S, C, Q, A가 최소 1개씩 존재
- 모든 A에 P가 2개 이상 (MECE)
- 형제 노드의 정렬 원칙이 결정되어있음
- 갭 목록이 비어있음

완성 시:
```
피라미드 완성. status: building → complete로 변경합니다.
다음 단계: /explain으로 문서화 | /prd로 실행 명세 | /go로 실행
```

### 규칙

- **노드 삭제 금지** — 재구성 시 연결이 끊긴 노드는 삭제하지 않고 `## 고아 노드` 섹션에 보관한다. 사용자가 대답한 모든 것은 유효했기에 나온 것. 사용자가 명시적으로 "삭제해"라고 말하기 전까지 유지한다. 고아 노드는 나중에 다른 곳에 연결되거나, 새 피라미드의 재료가 될 수 있다.
- **파일은 항상 최신 상태** — 매 턴 갱신. 대화가 끊겨도 파일에 진행 상태가 남아있다
- **AI가 판단을 먼저 밝힌다** — "이건 P(근거)로 보입니다" 식으로 분류를 제시하고 확인
- **강제 분류 금지** — 애매하면 물어본다
- **노드 ID는 순차** — S1, S2, C1, A1, P1, P2... 타입별 카운터
- **Mermaid 노드 라벨**: `TYPE_ID["타입: 내용 요약(20자 이내)"]` — 긴 내용은 노드 목록 테이블에
