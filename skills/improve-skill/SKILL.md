## 역할

너는 **스킬 ��계자**다. retro가 "무엇이 ��족한가"를 발견했다면, 이 스킬은 "어떻게 고칠 것인가"를 설계하고 실행한다.

## 강제력 위계

개선 수단을 선택할 때 이 위계를 따른다:

```
훅 (기계적 차단, 100%) > 스킬 (워크플로우 강제, ~80%) > CLAUDE.md/Memory (참고, ~50%)
```

- **기계적으로 잡을 수 있는 규칙** → 훅으로 구현 (regex, 파일 체크 등)
- **워크플로우/판단이 필요한 규칙** → 스킬 패치
- **맥락/배경 지식** → memory (최후 수단)

## Step 0: 입력 확인

retro 판정 테이블을 확인한다. 없으면 대화 맥락에서 개선 대상을 파악한다.

| 판정 | 대상 | 갭 설명 | 개선 방향 |
|------|------|---------|----------|
| 🔧 | ... | ... | ... |
| ⚙️ | ... | ... | ... |

🔧 = 스킬 패치, ⚙️ = 훅 추가/수정

## Step 1: 진단 — 스킬/훅 읽기

각 개선 대상에 대해:

1. **해당 스��/훅 파일을 읽는다**
2. **"이 갭을 방지하려면 스킬/훅의 어디에 뭘 추가/변경해야 하는가?"를 구체화한다**

진단 결과를 표로 정리:

| 대상 | 현재 상태 | 부족한 점 | 패치 내용 |
|------|----------|----------|----------|
| /go SKILL.md | verify 단계에 훅 테스트 없음 | 훅 파일 작성 후 검증 안 ��� | verify 체크리스트에 "훅 파일이면 단위 테스트" 추가 |

## Step 2: 훅화 가능성 판단

각 갭에 대해 **훅으로 기계�� 차단이 가능한지** 먼저 판단한다:

```
질문: "이 실수를 regex나 파일 체크로 감지할 수 있는가?"
  → Yes: 훅 구현 (⚙️)
  → No: 스킬 패치 (🔧)
```

### 훅 구현 시 체��리스트

- [ ] Pre or Post? — 차단이면 Pre, 피드백이면 Post
- [ ] sync or async? — 빠르면 sync, 느리면 async
- [ ] 제외 조건 — 테스트 파일, node_modules, 특정 폴더
- [ ] 에러 메시지 — 무엇이 잘못이고, 대안은 뭔지 명확히
- [ ] settings.json 등록

### 스킬 패치 시 체크리스트

- [ ] 어떤 Step��� 추가하는가?
- [ ] 기존 흐름을 깨지 않는가?
- [ ] 추가된 내용이 모든 상황에 적용되는가, 특정 조건에만인가?

## Step 3: 실행

진단 결과에 따라 파일을 수정한다.

**수정할 때 반드시 보여줄 것:**
- 상황: 어떤 갭이 발생했는가
- 원인: 스킬/훅의 어느 부분이 부족했는가
- 패치: 무엇을 어떻게 수정했는가
- 효과: 다음 세션에서 어떻게 달라지는가

## Step 4: 검증

- 훅 → 단위 테스트 (stdin에 JSON 주입, 결과 확인)
- 스킬 → 변경된 부분이 논리적으로 일관적인지 확인
