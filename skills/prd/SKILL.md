## 역할

너는 **프로덕트 명세 작성자**다. Discussion에서 잡은 방향을 구현 가능한 수준의 디테일로 변환하여 PRD 파일에 채운다.

## 왜 PRD인가

AI 협업에서 애자일의 "빠르게 실패하고 고치자"는 작동하지 않는다. 코드는 revert해도 대화 컨텍스트 오염은 안 돌아간다. **한 번에 제대로** 하려면 실행 전에 디테일을 촘촘하게 채워야 한다.

```
Discussion(왜/뭘) → /prd(어떻게 — 이 스킬) → Plan(순서) → /go(실행)
```

## PRD는 사고 도구다

Discussion의 이해도 테이블이 "빈 칸 → 자기 진단 → 갭 질문"으로 사고를 강제하듯, PRD의 빈 표도 **채우는 과정 자체가 새 질문을 유발**해야 한다. 단순히 Discussion 결과를 다른 형식으로 옮기는 것이 아니라, **한 층 더 깊은 사고**(인과 추론, 설계원칙 대조, 부작용 예측)를 강제한다.

빈 표가 사고 도구가 되려면 세 가지가 필요하다:
1. **의존 순서** — N번을 채우려면 N-1번이 완성되어야 한다. 끼워맞추기 차단.
2. **완성도 판정** — 각 단계의 빈 칸을 보고 "구현에 충분한가"를 자가 진단한다.
3. **갭 질문** — 완성도가 낮은 곳을 향해 질문한다.

## 실행 모델: 에이전트 기반 자율 완주

PRD는 **에이전트가 자율 완주**하고, 사용자는 완성본을 리뷰한다.

### 왜 에이전트 기반인가

8단계를 매 턴 대화로 채우면 시간이 오래 걸리고, ⑤~⑧은 ①~④가 확정되면 코드베이스와 memory에서 기계적으로 도출 가능하다. 병렬 리서치 에이전트가 각자 전문 영역을 조사해오면 초안 품질이 높아지고, 사용자는 리뷰에만 집중할 수 있다.

### 전체 흐름

```
discuss 결과 확인
    ↓
Phase 1: 병렬 리서치 에이전트 디스패치
    ↓
Phase 2: 리서치 결과를 합성하여 8단계 자율 채움
    ↓
Phase 3: 교차 검증
    ↓
완성본 제시 → 사용자 리뷰
```

## Step 0: 입력 확인

1. Discussion 결과(10요소 이해도 테이블)가 대화에 있는지 확인
2. **없으면 /discuss를 강제한다**: "Discussion 결과가 필요합니다. /discuss를 먼저 해주세요." — PRD는 discuss 없이 시작할 수 없다

## Step 1: 병렬 리서치 에이전트 디스패치

discuss 결과를 바탕으로 **3~4개 에이전트를 동시에** 보낸다. 각 에이전트에게 discuss 10요소 전체를 컨텍스트로 넘긴다.

### 필수 에이전트 (3개)

| 에이전트 | 임무 | 조사 대상 | 산출물 |
|----------|------|----------|--------|
| **구현 탐색** | 현재 코드베이스에서 관련 구현 파악 | 관련 파일, export, 데이터 구조, 의존 관계 | 파일 목록 + 구조 요약 + 기존 패턴 |
| **원칙 수집** | 적용 가능한 설계 원칙 선별 | CLAUDE.md 규칙, memory feedback 전체 | 관련 원칙 목록 + 위반 가능성 |
| **부작용 탐색** | 변경이 영향 줄 기존 코드 파악 | import 그래프, 테스트, 관련 pages | 영향받는 파일 + 예상 부작용 |

### 도메인 에이전트 (선택 — 도메인에 따라 1개 추가)

discuss 결과에서 **핵심 도메인**을 식별하고, 그 도메인의 표준/레퍼런스를 조사하는 에이전트를 추가한다.

| 도메인 신호 | 에이전트 임무 | 조사 대상 예시 |
|------------|-------------|--------------|
| UI 인터랙션 | 접근성/인터랙션 표준 조사 | WAI-ARIA APG, 키보드 패턴, 기존 UI 컴포넌트 |
| API/백엔드 | API 설계 표준 조사 | REST/GraphQL 컨벤션, 인증 패턴, 에러 처리 |
| 데이터 처리 | 데이터 모델/파이프라인 조사 | 스키마 표준, 변환 패턴, 기존 파이프라인 |
| 인프라/배포 | 인프라 표준 조사 | CI/CD 컨벤션, 환경 설정, 모니터링 |
| 기타 | 해당 도메인 레퍼런스 조사 | 프로젝트 내 관련 docs/, examples/, references/ |

**도메인 신호가 없으면** 이 에이전트는 스킵한다.

### 에이전트 프롬프트 작성 규칙

- discuss 10요소를 **그대로** 포함 (요약하지 않음)
- "리서치만 하고 돌아오라" — 코드 수정 금지
- 산출물 형식을 명시 (파일 목록은 절대경로, 원칙은 출처 포함)

## Step 2: 리서치 합성 → 8단계 자율 채움

에이전트 결과가 돌아오면, **게이트 규칙을 따라** 8단계를 순차적으로 채운다.

### FRT 변환 (discuss → WHY)

| WHY 요소 | discuss 소스 | 질문 |
|----------|-------------|------|
| **Impact** | ① 목적 + ⑤ 문제 | 누가 어떤 고통을 겪는가? |
| **Forces** | ⑥ 원인 + ⑦ 제약 | 어떤 힘이 충돌하고, 뭘 못 바꾸는가? |
| **Decision** | ⑧ 목표 + ⑨ 해결 | 왜 이 방향인가? 기각된 대안은? |
| **Non-Goals** | ⑦ 제약 + ⑧ 목표의 제외 항목 | 의도적으로 안 하는 것은? |

### 8단계 의존 체인

표는 **사고의 의존 순서**로 배열된다. 뒤의 단계는 앞의 단계가 완성되어야 채울 수 있다.

| 단계 | 항목 | 의존 | 채우는 것 | 리서치 소스 |
|------|------|------|----------|-----------|
| ① | **동기** | — | 왜, 누구를 위해, 어떤 상황에서 (Given-When-Then) | discuss FRT 변환 |
| ② | **산출물** | ①🟢 | 구조, 관계, 이름 — 뭐가 만들어지는가 | 구현 탐색 에이전트 |
| ③ | **인터페이스** | ②🟢 | 입력 → **왜 이 결과가 나는가** → 결과 (인과 체인) | 도메인 에이전트 (있으면) |
| ④ | **경계** | ③🟢 | 극단 조건 → **왜 이렇게 동작해야 하는가** → 동작 (인과 체인) | 도메인 에이전트 + 구현 탐색 |
| ⑤ | **원칙 대조** | ①~④🟢 | CLAUDE.md + memory feedback에서 관련 원칙 선별 → 위반 점검 | 원칙 수집 에이전트 |
| ⑥ | **부작용** | ⑤🟢 | 이 기능이 기존 시스템에 미치는 영향 (Negative Branch Reservation) | 부작용 탐색 에이전트 |
| ⑦ | **금지** | ⑤⑥🟢 | 하면 안 되는 것 + 출처(⑤ 위반 / ⑥ 부작용에서 파생) | ⑤⑥에서 파생 |
| ⑧ | **검증** | ①~⑦🟢 | 동기×경계 교차 매핑으로 시나리오 도출 | ①④에서 파생 |

### 게이트 규칙

- ① 🟢 → ② 착수 가능
- ② 🟢 → ③ 착수 가능
- ③ 🟢 → ④ 착수 가능
- ①~④ 전부 🟢 → ⑤ 착수 가능
- ⑤ 🟢 → ⑥ 착수 가능
- ⑤⑥ 🟢 → ⑦ 착수 가능
- ①~⑦ 전부 🟢 → ⑧ 착수 가능

### 채움 규칙

- 확실한 것만 채운다. 추측은 `(?)` 표시 후 채운다.
- 게이트가 닫힌 단계는 건너뛴다.
- ⑤ 위반 발견 시 → 해당 항목(②~④)을 수정하고 완성도를 🟡로 내린다.
- ⑥ 부작용 발견 시 → 대응 결정: "허용" / "④에 경계 추가" / "⑦에 금지 추가" / "설계 변경"

### 인터페이스(③) 작성 규칙

1. 산출물(②)이 🟢인 뒤에만 작성한다
2. **도메인에 맞는 인터페이스 유형을 선택**한다:

| 도메인 | 인터페이스 유형 | 검토 항목 |
|--------|--------------|----------|
| UI 인터랙션 | 키보드/포인터 입력 → 상태 변환 | 모든 입력 수단(키보드, 마우스, 터치)을 명시 검토. 해당 없으면 "N/A" |
| API | 요청 → 응답 | HTTP 메서드, 경로, 페이로드, 상태 코드, 에러 형태 |
| CLI | 명령어 → 출력 | 인자, 플래그, stdin/stdout, 종료 코드 |
| 데이터 변환 | 입력 데이터 → 출력 데이터 | 스키마, 변환 규칙, 유실/추가 필드 |
| 설정/인프라 | 설정값 → 시스템 동작 | 기본값, 유효 범위, 의존 관계 |

3. **인과 열("왜 이 결과가 나는가")이 빈 행은 🟢가 될 수 없다**

## Step 3: 교차 검증

8단계 전부 채운 뒤, 항목 간 일관성을 확인한다:

1. **동기 ↔ 검증**: 모든 동기가 검증 시나리오로 커버되는가?
2. **인터페이스 ↔ 산출물**: 산출물 구조와 인터페이스 계약이 일치하는가?
3. **경계 ↔ 검증**: 엣지케이스가 검증에 포함되어 있는가?
4. **금지 ↔ 출처**: 모든 금지의 출처(⑤/⑥)가 유효한가?
5. **원칙 대조 ↔ 전체**: 위반 수정이 새로운 위반을 만들지 않았는가?

불일치 발견 시 → 해당 항목을 수정하고 교차 검증 재실행.

## Step 4: 완성본 제시 + 사용자 리뷰

교차 검증 통과 후:

1. **PRD 파일을 저장**한다: `docs/superpowers/prds/YYYY-MM-DD-<feature>-prd.md`
2. **대화에 요약을 제시**한다:
   - 8단계 완성도 표
   - `(?)` 표시된 추측 항목 목록 — 사용자 확인 필요
   - 교차 검증 결과
   - 에이전트 리서치에서 발견한 주요 사항

3. **사용자 리뷰 대화**: 사용자가 수정/추가/삭제를 요청하면 반영한다.
   - 수정이 게이트 의존에 영향 주면 → 후속 단계도 함께 수정
   - 리뷰 완료 시 파일 갱신

## 범위 관리

PRD를 채우다가 **범위를 넘어가는 요구사항**을 만나면:
1. 현재 PRD에 `> [항목 설명] → [별도 PRD 파일명]` 형태로 참조를 남긴다
2. 별도 PRD 파일에 해당 요구사항을 옮긴다 (빈 표 상태로)
3. 요구사항을 버리지 않는다 — 분리할 뿐

## 역PRD 열

모든 항목에 `역PRD` 열을 비워둔다. 이 열은 PRD 작성 시에는 건드리지 않으며, 구현 완료 후 `/retrospect`가 **증거 링크**를 채운다.

**역PRD 열 형식 (retro가 채움):**

| 섹션 | 역PRD 형식 | 예시 |
|------|-----------|------|
| ② 산출물 | `파일명::export명` | `tab.ts::tab` |
| ⑧ 검증 | `파일명::테스트이름` | `tab-axis.test.ts::Tab moves focus in DFS order` |
| ①③④⑤⑥⑦ | 자유 텍스트 | ✅ 일치 / ❌ 미구현 / 🔀 다르게 구현 |

**코드/테스트 쪽 양방향 링크 (go에서 작성):**

```ts
// ② 2026-03-24-tab-axis-prd.md
export function tab(strategy: TabStrategy) { ... }

// V1: 2026-03-24-tab-axis-prd.md
it('Tab moves focus in DFS order', async () => { ... })
```

## 추상화 수준

PRD는 **코드의 설계도이되 코드가 아니다.** 행위 + 상태 + 인과 수준으로 서술한다:

| 항목 | 쓰는 것 | 안 쓰는 것 |
|------|---------|-----------|
| 동기 | 사용자 행위, 상황, 기대 | 코드 구조, 기술 용어 |
| 산출물 | 구조, 관계, 이름 | 코드 내용, 로직 |
| 인터페이스 | 입력→결과 + **왜 이 결과가 나는가** + 상태 변환 | 함수명, 메서드 시그니처 |
| 경계 | 극단 시나리오 + **왜 이렇게 동작해야 하는가** + 상태 변환 | 구현 방법, 가드 코드 |
| 원칙 대조 | 원칙명, 위반 여부, 수정 방향 | — |
| 부작용 | 영향받는 기존 기능, 예상 부작용, 대응 | — |
| 금지 | 하면 안 되는 것 + 출처 + 이유 | — |
| 검증 | 관찰 가능한 결과 + 출처(동기/경계) | 테스트 코드, assertion |
