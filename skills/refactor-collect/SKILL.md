---

## 역할

코드리뷰 세션에서 사용자가 언급하는 개선 요청을 들어, **이 프로젝트 전용 컨벤션**으로 추상화하여 `docs/2-areas/refactoring-conventions.md`에 누적한다.

일반적인 코드 스멜 감지(O(n) 탐색, 중복 코드 등)는 Claude가 이미 잘 하므로 수집 대상이 아니다. 수집 대상은 "이 프로젝트에서 어떻게 해야 한다"는 패턴이다.

## 동작 방식

### 세션 시작

`docs/2-areas/refactoring-conventions.md`가 있으면 로드해서 기존 컨벤션 목록을 파악한다.

### 사용자 피드백 처리

사용자가 코드 개선을 언급할 때마다:

1. **일반 코드 스멜인지 프로젝트 컨벤션인지 판단한다**
   - 일반: "이 함수가 너무 길다", "변수명이 불명확하다" → 수집하지 않고 그냥 처리
   - 프로젝트 컨벤션: "여기서는 ax()를 써야 해", "store command로 처리해야 해" → 수집

2. **특정 인스턴스를 컨벤션 문장으로 추상화한다**
   - 인스턴스: "`style={{ color: 'red' }}` 쓰지 마"
   - 컨벤션: "인라인 style={} 금지 — ax() 또는 module.css 토큰만 사용"

3. **기존 컨벤션과 중복 여부를 확인한다**
   - 중복: 기존 항목을 보강 (예시 추가, 표현 강화)
   - 신규: 새 항목으로 추가

4. **문서에 즉시 저장한다** — 대화 끝에 몰아서 저장하지 않는다

### 컨벤션 문장 형식

```
**[카테고리] 제목** — 한 줄 규칙
- 예: 나쁜 예 → 좋은 예
- 근거: 왜 이렇게 해야 하는가 (사용자가 말한 이유)
```

카테고리 예시: `[스타일]`, `[상태관리]`, `[컴포넌트]`, `[이벤트]`, `[타입]`

## 문서 구조

`docs/2-areas/refactoring-conventions.md` 누적 형식:

```markdown
# 리팩토링 컨벤션

> 코드리뷰 세션에서 추출한 프로젝트 전용 패턴. 일반 코드 스멜은 제외.

## [카테고리명]

**[제목]** — [한 줄 규칙]
- 예: `나쁜 예` → `좋은 예`
- 근거: [이유]

---
```

## 세션 중 출력

사용자가 개선을 언급할 때:
- 수집 대상이면: "📌 컨벤션으로 추가: **[제목]**" 한 줄로 알린다
- 수집 대상이 아니면: 그냥 처리하고 별도 언급하지 않는다

세션 끝에 누적된 컨벤션 수를 요약한다: "이번 세션 +N개, 누적 M개"
