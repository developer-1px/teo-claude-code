# improve-design — 안티패턴 탐지 → 수정 → 검증 루프

## 구조

2-actor 루프:
- **탐지자** (sonnet agent) — 스크린샷을 보고 안티패턴을 전부 찾는다
- **수정자** (메인) — 탐지 결과를 받아 코드를 수정한다

```
1. 촬영 — 대상 라우트 스크린샷 (전체 + 영역별 zoom)
2. 탐지 — sonnet agent에게 스크린샷을 주고 안티패턴 수집
3. 수정 — 탐지 결과의 안티패턴을 코드로 수정
4. 재촬영 + 재탐지 — 같은 agent에게 다시 던져서 해결/미해결 판정
5. 반복 — 미해결이 남으면 3으로. 최대 3회 또는 미해결 0개 시 종료
```

## 1단계: 촬영

screenshot.mjs로 대상 라우트를 찍는다.

```bash
# 전체 (light)
node scripts/screenshot.mjs /대상라우트 --light --wait 2000

# 전체 (dark)
node scripts/screenshot.mjs /대상라우트 --dark --wait 2000
```

전체 스크린샷을 본 뒤, puppeteer clip으로 영역별 zoom을 3곳 이상 찍는다.
- 헤더/바 영역
- 주요 콘텐츠 영역
- 입력/하단 영역
- 사이드 패널 (있으면)

스트리밍/애니메이션이 있는 페이지는 `--wait`을 충분히 늘린다.

## 2단계: 탐지

Agent 도구를 다음 설정으로 호출한다:
- `model: "sonnet"`
- `description: "Design anti-pattern detector"`

프롬프트:

```
너는 디자인 안티패턴 탐지기다.
아래 스크린샷들을 보고 디자인 안티패턴을 촘촘하게 전부 찾아라.

"안티패턴"이란: 누가 봐도 잘못인 것. 취향이 아니라 결함.

규칙:
- 체크리스트 없이 자유롭게 봐라. 눈에 보이는 것을 전부 의심해라.
- 영역별 zoom을 하나씩 열어서 빠짐없이 훑어라.
- 사소한 것도 전부 적어라. 과잉 탐지 > 과소 탐지.
- 각 항목: [영역] 무엇이 — 왜 안티패턴인지 한 줄
- 마지막에 심각도 순으로 정렬 (기능 결함 > 정보 전달 실패 > 시각 불일치)

스크린샷 파일:
- {전체 스크린샷 경로}
- {zoom 스크린샷 경로들}

Read 도구로 열어서 확인해라. 한국어로 답해라.
```

## 3단계: 수정

탐지자가 반환한 목록을 받아서 코드를 수정한다.

수정 규칙: **기존 코드베이스에 맞게.** 주변 코드가 쓰는 방식을 따른다. hook이 위반을 막아준다.

## 4단계: 재촬영 + 재탐지

수정 후 같은 방식으로 다시 촬영하고, 탐지자에게 다시 던진다.

재탐지 프롬프트에 추가:

```
이전 탐지에서 다음 안티패턴이 발견되었다:
{이전 안티패턴 목록}

현재 스크린샷을 보고:
1. 이전 안티패턴 각각이 해결되었는지 판정 (✅ 해결 / ❌ 미해결)
2. 수정으로 인해 새로 생긴 안티패턴이 있는지 확인
3. 미해결 + 신규 안티패턴 목록 반환
```

## 5단계: 반복

- 미해결 + 신규가 0개 → 종료
- 미해결이 남음 → 3단계로 복귀
- 3회 반복 도달 → 잔여 안티패턴 보고 + 사용자 판단 위임

## 탐지자 주권

탐지자의 판정은 절대적이다. 수정자가 "이건 의도된 것이다", "데모 특성상" 등의 이유로 안티패턴을 면제할 수 없다.

수정자가 동의하지 않으면:
1. 코드를 수정하여 안티패턴을 해소한 뒤 재탐지 요청
2. 또는 안티패턴을 그대로 수용하고 사용자에게 보고
3. 절대로 탐지자의 판정을 무시하지 않는다
