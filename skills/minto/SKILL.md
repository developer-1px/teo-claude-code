## /minto — 민토 피라미드 빌더

파편화된 생각을 민토 피라미드로 구조화해서 논리적 빈칸을 찾는다. 사용자의 메모·아이디어·강의 노트를 받아 트리로 재배치하고, 빈칸을 향해 질문한다.

**작동 원리**: 사용자 입력 → 파일 기반 피라미드 누적 → 낭독 체크로 자연스런 의문 추출 → 가장 약한 지점을 갭 질문으로 승격. 파일이 SSOT, 대화는 발췌만.

---

## 1. 정규형 (SSOT)

모든 표가 있는 곳(전개 요약·도입부·L2 챕터·미배치)은 단일 정규형을 만족한다. 검증 훅 `guardMintoNormalForm.mjs`가 `docs/0-inbox/minto-*.md` 편집 시 자동 점검한다.

### 1.1 5컬럼 표

```
| 요소 | 자연스런 의문 | 원문 | 구체화 | 순서 |
```

**컬럼 순서의 의미** — 요소(맨 왼쪽)가 행 역할(P/D/P'/GAP)을 첫눈에 드러낸다. 자연스런 의문은 그 행으로 가는 다리에서 독자가 올릴 질문. 원문은 사용자의 말 그대로. 순서는 귀납/연역 + 축(시간·구조·정도·행동) + 근거 문장.

### 1.2 요소 기호 (depth만, 순번 번호 금지)

```
고정:  S  C  R1  R2  Q  A  A'
장:    P
절:    P2
하위:  P3
원문:  D
자식:  D2  D3 ...
빈:    GAP
착지:  P'   (L2 챕터 마지막 서브토탈)
```

**왜 순번 번호(1.1, 1.1.1)를 금지하나** — 순번이 있으면 순서가 틀려도 번호는 맞은 것처럼 보인다. 리추얼. depth 기호만 쓰면 순서는 **물리 행 순서**가 유일 SSOT이 되어 오류가 즉시 드러난다. 형제가 여러 개면 모두 같은 기호(D/D/D)가 반복된다.

### 1.3 순서 컬럼 = 추론 종류 + 축 + 근거

```
형식: (귀납|연역) · (시간|구조|정도|행동) N (근거 한 줄)
예:   연역 · 삼단논법 (선언 — 헤딩 회수)
     연역 1 (전제 — 자연어 = 모호)
     연역 2 (소전제 — 모호 입력 → 비결정 출력 필연)
     귀납 · 정도 1 (개인 경험 — 가벼운 증거)
     귀납 · 정도 4 (자기평가 불능 — 도피처 차단)
```

두 차원을 병기해야 `연역인데 귀납처럼 쌓음` 같은 불일치가 드러난다. 연역이면 **삼단논법**(대전제 → 소전제 → 결론) 형태로 써야 한다. 귀납이면 증거 축(시간·구조·정도·행동)과 무게 순서를 명시한다.

### 1.4 L2 챕터 표 + 낭독 블록 — 템플릿 참조

L2 챕터는 P 샌드위치(`P → D×N → P'`) + 표 직후 낭독 blockquote가 정규형. 슬롯/규칙/예시는 **`references/l2-chapter-template.md`** 참조 (1.1 레퍼런스에서 추출한 고정 포맷).

핵심만:
- 첫 P 원문 = 헤딩 텍스트 그대로 (훅 면제). 마지막 P' = "그러므로 ~" 착지 완성형.
- 빈 D = 원문 `—`, 낭독에선 `**("의문")**`만 남김.
- 낭독 체크의 목적은 ✓/⚠ 채점이 아니라 청중이 올릴 자연스런 의문을 뽑는 것.

## 2. 파일 섹션 구조

독자 읽기 순서대로 배치한다. 전개(요약)가 도입부(상세)보다 먼저 와서 독자가 뼈대를 먼저 잡고 내려간다.

```markdown
---
title: ...
stage: phase1 | phase2 | done
updated: YYYY-MM-DD
---

## 의도
이 글이 말하려는 건 [X]다.

## 전개(요약)
[5컬럼 표: S C Q A + 장/절 2depth + A' 한 번에]

**도입부 낭독**: > ... (3블록 분리)
**전개 낭독**: > ...
**결론 낭독**: > ...

## 도입부 (상세)
[5컬럼 표 — S/S1/S2/C/C1~/R1/R2/Q/A]

> (낭독 블록)

# 1장. [액션 타이틀]

## 1.1 [헤딩 = 챕터 결론 단정문]
[5컬럼 표 — P/D×N/P' 샌드위치]

> (낭독 블록)

## 1.2 ...

# 2장. ...

## 마무리
- **한 줄**: ...
- **A' (N점 착지)**: ...

## 미배치
[5컬럼 표 — 삭제하지 않은 노드들 보존]
```

### 2.1 전개(요약)의 해상도

전개(요약) 표는 **장+절 2depth까지** 필수. 장만 6행으로 압축하지 말 것 — 장 결론이 절들에서 실제 귀납되는지 검증할 수 없다. 2depth가 빌드업 검증의 최소 해상도.

### 2.2 전개↔P축 동기화 (절대 규칙)

전개(요약) 표의 각 행과 본장 L2 챕터의 P 축은 **항상 동기화**되어야 한다. 한 쪽이 바뀌면 다른 쪽도 같은 Edit에서 맞춰 바꾼다.

3점 정합성:

```
L2 챕터 헤딩 (## N.M ...)  ←→  본장 L2 챕터 P 행 원문  ←→  전개 요약 P2 행 원문
```

- 세 자리는 **의미상 같은 단정문**이어야 한다. 완전 동일하지 않더라도 독자가 "다른 주장"으로 느끼면 불일치.
- 가장 단순한 운용: 본장 L2 챕터 P 행 = 헤딩 그대로. 전개 요약 P2 = 본장 P 행 그대로 (혹은 한 문장 요약).
- **불일치 발견 시 즉시 수정.** "다음 턴에 고치자" 금지 — 사용자가 같은 지점을 두 번 읽을 때 다른 이야기로 들린다.

2장·3장 P 격 산문(헤딩 직후 본장 P 단락)도 전개 요약 장 P 행과 동기화: 산문의 핵심 주장이 전개 요약 P 행 원문과 같아야 한다.

### 2.3 전개 낭독 3블록 분리

전개(요약)의 낭독은 도입부 / 전개 / 결론 세 개의 독립 `>` 블록으로 분리한다. 기능 단위가 다른 구간을 한 호흡으로 묶으면 의문 추출이 뭉개진다.

## 3. Bottom-up 루프

사용자가 내용을 주면 파일을 만들어 트리로 재배치하며 시작한다.

### 3.1 매 턴 수행

1. **파일 로드** — 기존 `docs/0-inbox/minto-{slug}.md`가 있으면 Read. 없으면 새로 생성.
2. **의도 파악** — "이 글이 말하려는 건 [X]다"를 한 문단으로 써서 사용자 교정 받음.
3. **피라미드 재배치** — 트리 재배치는 **LLM 자율 역할**. 사용자에게 "어디에 둘까요?"를 묻지 말고 먼저 옮기고 "이렇게 했는데 어때요?"로 보고.
4. **파일 저장** — Edit/Write로 갱신. 대규모 변경은 Sonnet 위임 (아래 4.3).
5. **낭독 체크** — 작업한 섹션의 낭독 블록을 읽고 각 다리에서 자연스런 의문을 추출 → 표의 의문 컬럼에 박음.
6. **갭 질문** — 낭독 의문 중 가장 날카로운 것 1개를 갭 질문으로 승격 (아래 3.3).

### 3.2 Phase 구조

- **Phase 1 — 의도 + 전개(요약) 뼈대**: SCQA + 장 후보 나열. A'는 도입부 A에서 확장.
- **Phase 1.5 — 그루핑 게이트**: 어느 부모든 직계 자식 ≥7이면 Phase 2로 가지 말고 먼저 MECE 축으로 2~4개 그룹 묶기. L2 챕터 절 행 ≥7은 훅이 fail로 블록.
- **Phase 2 — 챕터 상세**: 각 L2 챕터를 P 샌드위치 + D 근거로 채운다. 추론 종류(귀납/연역)를 먼저 정하고 그에 맞는 형식(삼단논법/증거 사슬)으로 서술.
- **Phase 3 — 미배치 재수용**: 남은 파편은 삭제 금지, 미배치 표에 보존하거나 근거/수사 장치로 승격.

### 3.3 갭 질문 = 매 턴 마지막 출력 (절대 규칙)

**매 턴 마지막 출력은 갭 질문이어야 한다. 예외 없음.** 즉석 작성 금지, 낭독 블록의 의문에서 승격한다 (저자 편향 차단).

**역할 — LLM은 인터뷰어**: 사용자는 글쓴이, LLM은 독자 시점에서 질문을 던지는 인터뷰어. 사용자에게 트리 구조를 결정시키지 마라 ("이렇게 묶을까요, 저렇게 묶을까요?"). 트리는 LLM이 자율로 옮기고 보고만 한다. 갭 질문은 항상 **콘텐츠 질문** — "독자가 X라 물으면 뭐라 답하실 건가요?".

**갭 질문 → 자연스런 의문 컬럼 영속화 (절대 규칙)**: 매 턴 던진 갭 질문은 그 자리에서 끝나면 안 된다. **트리 재배치 시점에 해당 행의 `자연스런 의문` 컬럼에 즉시 박는다.** 어느 행에 박나 — 그 의문이 다리 역할을 하는 행 (사용자 답변이 새 D로 들어가면 그 새 D 행, 기존 행에 추가 정보가 박히면 그 행). 박지 않으면 다음 세션에서 사라진다.

박는 절차:
1. 갭 질문을 던진다 → 사용자가 답한다 → 답을 새 D 행 또는 기존 행에 박는다 (트리 재배치)
2. **같은 Edit에서** 그 행의 `자연스런 의문` 컬럼에 방금 던진 갭 질문을 박는다 (`—`를 질문 텍스트로 교체)
3. 갭 질문이 LLM 자율 연결로 해소된 경우(트리에 답이 이미 있어서 사용자에게 묻지 않은 경우)도 마찬가지 — 연결 마크를 박는 행의 `자연스런 의문` 컬럼에 그 의문을 박는다

이로써 트리 자체가 "어디에서 독자가 멈춰서 무엇을 물었는지"의 기록이 되고, 다음 세션이 그 의문 목록 위에서 이어갈 수 있다.

3단 구조(반영 → 보고 → 갭 질문) + 맥락 발췌 표 + 의문 톤/금지 규칙 = **`references/gap-question-format.md`** 참조. 이 포맷이 곧 매 턴 대화 출력 포맷이다.

## 4. 공통 규칙

### 4.1 원문 최대 보존

- **제1원칙**: 사용자의 원문을 정제/재작성하지 않는다. 표·도입부·P·D 모두 그대로.
- **파편 → 습니다체 완성 문장** 변환만 허용. "열받더라고" → "열받았습니다". 단어 교체·주어 삽입·부연 추가 금지.
- **출처·수치 두 갈래**:
  - **LLM 학습 내 사실** (이미 머리에 있는 출처·수치·메커니즘): D 작성 시점에 즉시 박는다. 되묻지 않음. 예 — Stack Overflow Survey, SWE-bench Verified, Lost in the Middle, Claude Code 훅 구조 등 LLM이 알고 있는 것.
  - **인터넷 검색 필요한 사실** (확실치 않거나 최신 수치 등): 박기 전에 "이 사실을 인터넷에서 찾아서 넣을까요?"라고 사용자에게 물어보고 동의 후에 검색·삽입한다. LLM이 추측으로 박는 건 금지.

### 4.2 트리 재배치는 LLM 자율 역할

사용자에게 "어디?/어떻게?"를 묻는 것은 판단 회피다. 먼저 배치하고 "이렇게 했는데 어때요?"로 보고. 틀리면 사용자가 고친다. 순서 변경, 그루핑, 챕터 swap, P 샌드위치 적용, 숨은 전제 D 추가 — 모두 LLM이 먼저 실행.

### 4.3 대규모 변경은 Sonnet 위임

컨텍스트가 빡빡해지면 (한 턴에 Edit 4+ 예상, 표 전체 재구성, 그루핑 여러 P 동시) **Sonnet 에이전트에게 위임**. Opus는 분석·구조·갭 질문에 집중하고 Sonnet은 텍스트 패치를 실행.

**위임 프롬프트 필수 요소**:
1. 대상 파일 절대경로
2. 변경 명세 (구조)
3. Opus가 작성한 새 텍스트 본문 (그대로 박을 것)
4. "원문 정제 금지" 같은 minto 규칙 명시
5. "변경 후 git diff 줄 수만 보고하라"

**위임 금지**:
- 새 원문 작성 (정제 위험)
- 갭 질문 작성 (저자 편향 우회 실패)

### 4.4 사용자 명시 삭제 외엔 삭제 금지

갈 곳 없는 파편은 미배치 표로. "너무 중복이라 압축"은 사용자가 명시할 때만.

## 6. 파일 생명주기

1. **첫 턴**: `docs/0-inbox/minto-{slug}.md` 생성. slug는 주제에서 추출.
2. **매 턴 시작**: Read로 파일 로드.
3. **매 턴 끝**: Edit/Write로 갱신. 파일이 SSOT.
