## /minto — 민토 피라미드

파편화된 생각을 민토 피라미드로 구조화하여 논리적 빈칸을 찾는다.

---

### 모드 판별 (첫 턴)

```
사용자 입력에 내용(메모, 파편, 텍스트)이 있는가?
  → YES: Bottom-up — 내용을 트리로 재배치하며 시작
  → NO:  Top-down — (미구현)
```

이후 턴은 모드 무관하게 동일한 루프를 탄다.

---

### 정규형 (산출물 SSOT) — 절대 규칙

산출물 파일은 **단일 정규형**을 강제로 만족해야 한다. 표가 있는 모든 곳(도입부·L2 챕터·미배치·결론후보·수사장치)이 동일한 **7컬럼 공통 스키마**를 따른다 — 엔티티처럼 통일.

#### 위계 = 마크다운 헤딩

```markdown
## 도입부                          ← L0
| 접속사 | 낭독 | 원문 | 구체화 | 순서 | 위치 근거 | 요소 |
|--------|------|------|--------|------|----------|------|
| — | — | ... | 원칙 | 도입 1 | 공통 지반 | S |
| — | — | ... | 사례 | 도입 2 | S를 흔드는 진짜 변화 | C |
| — | — | ... | ... | ... | ... | R1 |
| — | — | ... | ... | ... | ... | R2 |
| — | — | ... | — | ... | R1/R2 갭에서 자연 생성 | Q |
| — | — | ... | — | ... | Q 직답 | A |

# 1장. [액션 타이틀 — 결론 문장]    ← L1

## 1.1 [액션 타이틀 — 챕터 결론]    ← L2

| 접속사 | 낭독 | 원문 | 구체화 | 순서 | 위치 근거 | 요소 |
|--------|------|------|--------|------|----------|------|
| — | — | (원문 그대로) | 원칙 | 시간 1 | 본질적 갭 — 인과의 출발 | 1.1.1 |
| 그래서 | [...마지막 한 절] 그래서 [...이 행 첫 한 절] | ... | 수치 | 시간 2 | 구조적 원인 | 1.1.2 |
| — | — | (챕터 결론 재진술) | 원칙+수치 | 그루핑: 시간 | 축=인과 사슬 / MECE ✓ / 액션 타이틀 ✓ | → 1.1 |

## 1.2 [...]
...

# 2장. ...

## 마무리                          ← L4
- **한 줄**: ...
- **A'** (4점 착지): ...

## 결론 후보                       ← 같은 7컬럼
## 수사 장치                       ← 같은 7컬럼

## 미배치                          ← 안전망, 같은 7컬럼
```

#### 7컬럼 공통 스키마 + 셀 형식 강제

체크리스트(☐ 예/아니오)는 자기평가가 후하다. 7컬럼은 **빈칸 강제 게이트**다 — 셀 형식이 닫혀 있어야 LLM이 거짓말을 못 하고, 채운 내용 자체가 자동 검증 입력이 된다.

**컬럼 순서가 게이트다.** LLM은 왼쪽→오른쪽으로 채우기 때문에, 가장 왼쪽에 둔 컬럼이 가장 먼저 결정된다. 번호(요소)를 가장 왼쪽에 두면 자리가 사전 결정되고 모든 컬럼은 사후 라벨이 된다 — **이것이 위치 근거가 리추얼이 된 이유**다. 새 정규형은 **접속사를 가장 왼쪽**에 둬서 LLM이 행을 시작하는 순간 강제로 이전 행과의 연결을 결정하게 만든다. 번호는 가장 오른쪽으로 보내져 결과로만 부여된다.

정규형 = **6컬럼 표 + 챕터별 낭독 블록**. 낭독은 표 안의 셀이 아니라 표 다음에 오는 별도 줄글 블록(인용 `>`)이다. 표 안에 두면 이전 행 어절과 이 행 어절이 양쪽 셀에 중복돼 가독성이 무너진다. 줄글 블록으로 빼면 사슬이 한 호흡에 읽혀서 어색함이 즉시 드러난다.

| # | 컬럼 | 형식 | 의미 | 자동 검증 |
|---|------|------|------|----------|
| **1** | **접속사** | 닫힌 12 어휘. 인과(`그래서·따라서·그러므로·결국·마침내`) / 부연(`즉·다시 말해·게다가·또한`) / 반박(`그런데·그러나·하지만·그런데도`) / 전환(`한편·반면`) / 예시(`예를 들어·가령`). 첫 행만 `—` 허용. **어울림이 애매하면 `?` suffix** (예: `따라서?`, `그런데?`) — LLM이 사슬을 확신하지 못한 자리를 자기 노출. 검사 17(매핑) 면제, 검사 18로 warn 출력 | **이전 행과 이 행을 잇는 다리**. 안 떠오르면 사슬이 끊어졌다는 결정적 증거 — 행을 못 적음. `?`는 "다리는 놓았는데 어색하다"는 신호 — 사용자가 즉시 점검할 자리 | **검사 13** 빈칸 fail · **검사 14** 닫힌 어휘 외 fail · **검사 18** `?` 표시 warn |
| **2** | **원문** | **습니다체 완전 문장 (≥18자, 종결어미 `습니다/입니다/십시오/…` 필수).** 명사·동사·감정 표현 1:1 보존, 채우기 단어만 제거. 메모/제목/키워드 형태 금지 | 강연자의 말을 그대로 옮긴 한 문장 | 표 사이 중복 금지 · **검사 11** (종결어미·길이·파편) |
| **3** | **구체화** | `(원칙\|사례\|수치\|인용)([+](원칙\|사례\|수치\|인용))*` | 원문 종류 분류 | A4·A5 |
| **4** | **순서** | `(시간\|구조\|정도) \d+ \(이유\)` 또는 `도입 \d+` 또는 `그루핑: 시간\|구조\|정도 \(축\)` | 형제 내 위치 + 기준 | C2·C3·C4 (괄호 안 기준 명시 강제) |
| **5** | **위치 근거** | `\[어휘\] 설명`. 어휘 = **닫힌 12개** (`전제·정의·증거·보강·반박·귀결·문제·행동·상황·훅·엔딩·맥락`) | "왜 이 자리인가" + 역할 prefix. **접속사 카테고리와 일관해야 함** | **검사 17** 접속사 카테고리 ↔ prefix 매핑 fail · **검사 9** prefix 누락 |
| **6** | **요소** | `S\|C\|R1\|R2\|Q\|A` 또는 `\d+(\.\d+)*` 또는 `→ \d+(\.\d+)*` | 행 번호. **결과로만 부여**, 어떤 결정도 하지 않음 | — |

##### 낭독 블록 (표 다음에 오는 별도 줄글)

L2 챕터 표 직후에 인용블록(`>`)으로 사슬을 한 호흡에 적는다. 형식:

```markdown
> [1.1.1 원문]. **([1.1.2 접속사])** [1.1.2 원문]. **([1.1.3 접속사])** [1.1.3 원문]. **([1.1.4 접속사])** [1.1.4 원문].

**어울림 검토**:
- ([접속사1]) [N.1] → [N.2]: [한 줄 평가] — ✓ 또는 ⚠
- ([접속사2]) [N.2] → [N.3]: [한 줄 평가] — ✓ 또는 ⚠
- ([접속사3]) [N.3] → [N.4]: [한 줄 평가] — ✓ 또는 ⚠
```

검사 19가 챕터 표 다음 인용블록 존재를 확인한다. **어울림 검토 항목별** ✓/⚠은 LLM이 한 줄씩 판정 — "전체 ✓" 같은 막연한 자기평가가 아니라 다리 하나하나에 대한 한 줄 판정이라 리추얼화 어려움.

##### 채우기 순서 (왼쪽→오른쪽 = LLM 사고 순서)

1. **접속사** — 이전 행 마지막 문장에서 이 행으로 가는 다리를 닫힌 12 어휘에서 고른다. **안 떠오르면 멈춤.** 사슬이 끊어졌다는 신호다. 첫 행만 면제.
2. **원문** — 이전 행과 접속사로 자연스럽게 이어지도록 완전 습니다체 문장을 적는다. 낭독 블록에서 한 호흡에 읽었을 때 어색하면 이 자리에서 다시 잡는다.
3. **구체화** — 원칙/사례/수치/인용 분류.
4. **순서** — 시간/구조/정도 + 근거.
5. **위치 근거** — `[어휘]` prefix. 접속사 카테고리와 자동 일관성 검사 (검사 17).
6. **요소(번호)** — 가장 마지막. 번호는 결과일 뿐.
7. **(표 완성 후) 낭독 블록 작성** — 챕터 표 다음에 인용블록으로 사슬을 적고 어울림 검토를 항목별로 ✓/⚠ 판정.

##### 접속사 카테고리 ↔ 위치 근거 prefix 매핑

| 접속사 카테고리 | 어휘 | 호환 prefix |
|---|---|---|
| 인과 | 그래서·따라서·그러므로·결국·마침내 | [귀결] [증거] [행동] [엔딩] |
| 부연 | 즉·다시 말해·게다가·더구나·또한 | [전제] [정의] [증거] [보강] [상황] [맥락] |
| 반박 | 그런데·그러나·하지만·그렇지만·그런데도 | [반박] [문제] |
| 전환 | 한편·반면 | [전제] [정의] [상황] [문제] [훅] [맥락] |
| 예시 | 예를 들어·가령 | [증거] [보강] [훅] |

##### 닫힌 어휘 12개 (위치 근거 prefix)

```
[전제]  — 논증의 출발점, 다른 모든 근거의 뿌리
[정의]  — 개념을 정의하는 진술
[증거]  — 정의·주장을 떠받치는 사례·수치·인용
[보강]  — 약한 증거에 덧붙이는 추가 증거
[반박]  — 청중의 예상 반론을 미리 차단
[귀결]  — 인과의 끝, 결과
[문제]  — 부정적 현상·갭
[행동]  — 청중에게 시키는 동작
[상황]  — 청중이 알아야 할 배경
[훅]    — 청중 주의를 끄는 수사 장치
[엔딩]  — 마무리 한 줄
[맥락]  — 다른 행에 종속된 보충 (예: D0-증1)
```

서브토탈 행(`→ X.Y`) 위치 근거는 `[정의] 축=... / MECE ✓ / 액션 ✓` 형식.

##### 셀 형식이 자동 검증하는 9섹션 항목

- **A4·A5** 동종성/근거 종류 — 구체화 컬럼이 종류를 명시하므로 한눈에 보임
- **A6·D4** 액션 타이틀 / 카테고리 라벨 — 위치 근거 prefix가 12 어휘로 닫혀 있어 `...들` 같은 라벨 자동 차단
- **B5** 연역 ≤3단 — (폐기됨, 위치 근거 prefix로 추론 흡수)
- **B8** 그룹 내 흐름 섞임 — 같은 L2 챕터의 절 행 접속사 카테고리가 2개 이상이면 warn
- **C2·C3·C4** 순서 기준 명시 — 순서 셀 괄호 안 누락이면 warn
- **D3** So What — 위치 근거 prefix가 [정의·귀결·증거...] 중 하나여야 하므로 자유 서술 회피 불가
- **F1** 행동 vs 상황 — `[행동]/[상황]` prefix로 자동 분류

#### 강제 6규칙

1. **원문은 L2 챕터 표 또는 미배치/결론후보/수사장치 표에만** 산다 — L0 도입부 표 제외 시, L1 장은 헤딩 텍스트만, L4 마무리는 A'/엔딩 줄글만. 절 원문은 L2에 또는 부록 표에 박힌다
2. **7 임계는 모든 부모-자식 관계에** — L2 챕터 절 행, **L1 장 안의 L2 챕터 수, 책 전체 L1 장 수** 셋 다 ≥7이면 그루핑/분할 강제. 서브토탈 행(`→X.Y`)은 카운트 제외. 도입부 표(S/C/R1/R2/Q/A)는 6요소 고정이라 자동 통과
3. **모든 표는 정확히 7컬럼** — 컬럼 수가 다르면 훅 fail
4. **결론 문장은 액션 타이틀** — 카테고리 라벨(`...들`, `...에 관한`, `기타`) 금지. 액션·결과·판단 형태여야 함 (So What 테스트)
5. **검증 메타·이력은 본문 금지 (휘발)** — `<!-- Grouping -->`, `검증 (N차)`, `← 원 P*에서 이동`, MECE 검증 표, 스토리 게이트 낭독 모두 본문에서 제거. 매 턴 대화 출력에만
6. **사용자 명시 삭제 외엔 삭제 금지** — 갈 곳 없는 원문은 미배치 표로

| # | 민토 요소 | 어디서 강제되나 |
|---|----------|----------------|
| 1 | 그루핑 라벨 테스트 | 서브토탈 행 위치 근거의 "축=..." + 헤딩 액션 타이틀 |

#### Phase 1.5: 전개 검토 게이트 (절 채우기 전 필수)

도입부(L0)가 통과되면 **절 원문을 채우기 전**에 L1+L2 청사진 검토를 강제한다. 산출물 상단에 `## 전개` 섹션을 두고, L1 장과 L2 챕터의 결론·순서·추론·MECE만 먼저 잡는다.

**전개 표도 검사 11 대상이다.** L1 장 결론, L2 챕터 결론 모두 **습니다체 완전 문장**으로 적어야 한다. 명사형 제목(`바이브 코딩만으론 안 된다`, `하네스의 4가지 도구`, `남는 능력 = 설명·결정·분할정복`)은 금지. 완전 문장(`바이브 코딩만으로는 파트너 수준의 주니어가 될 수 없습니다.`, `하네스에는 컨텍스트·스킬·에이전트·훅 네 가지 도구가 있고 강제력 낮은 순서대로 쌓입니다.`, `AI 시대 개발자에게 남는 능력은 설명력·결정력·분할정복 세 축입니다.`)으로 적어야 한다. 헤딩 텍스트(`# 1장. ...`)는 짧은 제목이 허용되지만, 표 셀의 결론은 완전 문장이어야 한다.

```markdown
## 전개

| 접속사 | 낭독 | 원문(결론) | 구체화 | 순서 | 위치 근거 | 요소 |
|--------|------|------------|--------|------|----------|------|
| — | — | ... | — | 시간 1 | 문제 진단 | 1장 |
| 즉 | [...마지막 한 절] 즉 [...이 행 첫 한 절] | ... | — | 정도 1 | ... | 1.1 |
| 그래서 | [...마지막 한 절] 그래서 [...이 행 첫 한 절] | ... | — | 정도 2 | ... | 1.2 |
| — | — | ... | — | 그루핑: 정도 | 축=귀책 무게 / MECE ✓ / 액션 ✓ | → 1장 |
| 한편 | [...마지막 한 절] 한편 [...이 행 첫 한 절] | ... | ... | 시간 2 | ... | 2장 |
```

**게이트 통과 조건**:
- 모든 L1·L2 결론이 액션 타이틀 (카테고리 라벨 금지)
- 7 임계 통과 (장 수, 챕터 수)
- 각 부모마다 서브토탈 행에 MECE 판정
- 사용자 명시 승인

게이트를 통과해야 Phase 2(절 채우기)로 진입한다. 통과 전에 L2 챕터 표에 절 행을 추가하면 청사진과 어긋날 위험.

#### 정적 검증 훅

`docs/0-inbox/minto-*.md` Edit/Write 시 PostToolUse 훅 `guardMintoNormalForm.mjs`가 검사:

### Bottom-up 모드

사용자가 내용을 주면 트리를 만들어서 시작한다. **모든 결과물은 위 정규형을 따른다.**

#### 매 턴 수행

```
0. 파일 로드 — 기존 파일이 있으면 Read로 읽는다. 없으면 1에서 새로 만든다.
1. 의도 파악 — "이 글이 말하려는 건 [X]다"를 먼저 쓴다
2. 피라미드 재배치 — 도입부 → 전개 → 결론 초안, P별 표
3. 파일 저장 — Edit/Write로 파일 갱신 (또는 대규모 수정은 Sonnet 에이전트 위임 — 아래 규칙 참조)
4. 갭 질문 — 매 턴 마지막은 반드시 빈틈을 향한 반문으로 끝낸다
```

##### 매 턴 마지막 = 갭 질문 (절대 규칙)

**모든 턴의 마지막 출력은 갭 질문이어야 한다.** 예외 없음.

- ❌ "다음 갭 질문 가시겠어요?" — 사용자에게 진행 허락을 묻는 것은 갭 질문이 아니다. 판단 회피.
- ❌ "이 방향으로 가시겠어요?" — 메타 합의 질문도 갭 질문이 아니다.
- ❌ "더 다듬을까요?" — 작업 분량 협상.
- ✅ "청중이 'X라고 하면 어떻게 답하실 건가요?'" — 피라미드 내용을 향한 반문.
- ✅ "D5의 근거가 경험뿐인데, 같은 결론을 반박하는 사례가 있나요?" — 빌드업 약점을 향한 질문.

만약 모든 노드가 완성이고 정말 더 물을 게 없다면, **수사 장치 / 결론 후보 / 도입부 검증** 중 가장 약한 곳을 찾아 묻는다. 그것도 없다면 **A를 한 줄 더 단단하게 만들 수 있는 반박 시나리오**를 묻는다. "끝났다"는 판정은 사용자만 한다.

##### 파일 수정 위임 규칙 (컨텍스트 절약)

대화 컨텍스트가 빡빡해지면(긴 표 여러 개를 동시에 편집할 때, 또는 한 턴에 5+ Edit 호출이 예상될 때) **Sonnet 에이전트에게 파일 수정을 위임**한다. Opus는 분석·구조·갭 질문에 집중하고, Sonnet은 텍스트 패치를 실행한다.

**위임 트리거**:
- 한 턴에 Edit 호출이 4개 이상 예상될 때
- 표 전체 재구성처럼 긴 old_string/new_string이 필요할 때
- 그루핑 게이트가 여러 P에서 동시에 발동했을 때

**위임 방법**:
```
Agent 호출 (subagent_type: general-purpose, model: sonnet)
프롬프트 구성:
1. 대상 파일 절대경로
2. 변경 사항 구체 명세 — 어느 섹션을 어떻게 바꿀지 (구조)
3. Opus가 작성한 새 텍스트 본문 (그대로 박을 것)
4. "원문 D는 보존, 정제 금지" 같은 minto 규칙 명시
5. "변경 후 git diff 줄 수만 보고하라" — 파일 본문 다시 받지 않음
```

**위임하지 않는 경우**:
- Edit 1~3개 — Opus가 직접 한다
- 새 D 텍스트 작성 — 정제 위험이 있어 항상 Opus가 직접
- 갭 질문 작성 — 절대 위임 금지

#### 1. 의도 파악

사용자 입력을 읽고 **"이 글이 말하려는 건 [X]다"**를 한 문단으로 쓴다. 사용자가 교정하면 반영한다. 의도를 모르면 피라미드도 질문도 엉뚱해진다.

#### 2. 피라미드 재배치

##### Phase 1: 뼈대 — 도입부 / 전개 / 결론

먼저 큰 틀을 잡는다. 세부 D는 나중에.

```
도입부:
- S: 독자가 이미 아는 사실 (+S1, S2... 하위 파편 가능)
- C: 무엇이 변했거나 문제인가 (+C1, C2... 하위 파편 가능)
- R1: 원하지 않는 결과 (+R1-a, R1-b... 하위 파편 가능)
- R2: 원하는 결과 (+R2-a, R2-b... 하위 파편 가능)
- Q: S+C(+R1/R2 갭)에서 떠오르는 질문
- A: Q에 대한 한 문장 직답

전개:
- P1: [결론을 뒷받침하는 첫 번째 근거]
  - P1-1, P1-2... (서브P — P가 크면 그루핑)
- P2: [두 번째 근거]
- P3: [세 번째 근거]
(P 레벨 순서 타입 + 근거를 명시)

결론:
- A': 전개를 거친 뒤의 풍부해진 핵심 메시지
```

**도입부 확장성**: S/C/R1/R2도 P/D처럼 하위 파편을 가질 수 있다. 한 문장으로 못 담으면 하위 노드를 만든다. 단 **Q와 A는 반드시 한 문장**이다 — 질문과 직답은 파편화되면 독자가 받을 수 없다.

도입부 표기 예시 (C가 풍부한 경우):

```markdown
| 요소 | 원문 |
|------|------|
| S | 바이브 코딩이 기정사실화됐다 |
| **C** | **채용 구조가 근본적으로 바뀌고 있다** |
| C1 | 이미 채용이 동결되고 있고 동결되는 이유가 그것이다 |
| C2 | 주니어를 뽑을 비용으로 시니어에게 AI를 결제해서 준다 |
| C3 | 시니어도 주니어를 교육할 바에 AI를 쓰고 싶다고 한다 |
| C4 | 코딩 노동력이 필요해서 사람을 뽑는 건 아니다 — 파트너 수준이 아니면 안 뽑는다 |
| R1 | ... |
| R2 | ... |
| Q | ... |
| A | ... |
```

**A / A'**: 도입부 A는 Q에 대한 직답(한 문장). 결론 A'는 전개(P1~Pn)를 거친 뒤 독자가 다르게 읽을 수 있는 풍부해진 버전. 같은 주장이지만 구체성과 무게가 다르다.
```

**R1/R2**: 문제 정의에 "원하지 않는 결과"와 "원하는 결과"를 명시한다. 갭이 Q를 만든다. 둘 다 없으면 빈칸.

##### 도입부 검증 — P로 넘어가기 전 필수

민토 원서의 핵심 주장: **"독자가 Q를 품고 있어야 A가 들어온다."** 도입부가 검증되지 않으면 전개 전체가 허공을 친다. P1 시작 전에 아래 체크리스트를 모두 통과해야 한다.

| 체크 | 판정 질문 | 실패 신호 |
|------|----------|----------|
| **S는 독자가 이미 아는가?** | 청중이 "아 맞아"로 받아들일 수 있는 사실인가? | 새 정보·주장·논쟁거리가 들어가 있으면 실패. S는 논쟁 없는 공통 지반이어야 한다 |
| **C는 진짜 변화인가?** | 이 Context가 없었으면 Q가 떠오르지 않았을 만큼 S를 흔드는가? | S와 무관하거나 평범한 배경 설명이면 실패 |
| **R1/R2 갭이 Q를 만드는가?** | R1(현재 싫은 것)과 R2(원하는 것)의 거리가 명확해서 독자가 "그럼 어떻게?"를 떠올리는가? | 둘 중 하나가 빠지거나 모호하면 실패. 갭이 Q의 연료다 |
| **Q는 청중이 실제로 떠올릴 질문인가?** | S+C+R을 읽고 청중 머릿속에서 자연스럽게 올라오는 질문인가, 아니면 화자가 하고 싶은 대답에서 역산한 질문인가? | 역산된 Q면 실패. 청중 언어로 다시 써야 한다 |
| **A는 Q에 직답인가?** | Q에 한 문장으로 답하는가? 옆으로 새지 않는가? | "음 그건 말이죠..." 식이면 실패. 주어-서술어로 Q에 직답 |
| **A는 P1~Pn이 지지할 수 있는가?** | 전개의 P들이 A를 귀납적으로 떠받칠 수 있는 주장인가? | P들과 무관한 주장이면 도입부 A가 아니라 다른 글의 결론이다 |
| **스토리 게이트 (필수)** | S→C→R→Q→A를 순서대로 읽었을 때 하나의 이야기로 흘러가는가? 각 요소가 다음 요소를 "필연적으로" 불러오는가? | 항목들이 병렬 선언문처럼 덜컥덜컥 끊기면 실패. 독자가 "그래서?" "그런데?" "그럼 어떻게?"를 스스로 떠올리며 다음 줄로 내려가야 한다 |

**7개 중 하나라도 fail이면 Phase 2로 넘어가지 말고 도입부 질문부터 한다.** 도입부가 흔들리면 P 전체가 흔들린다.

##### 스토리 게이트 낭독 테스트

S→C→R→Q→A를 다음 접속사로 연결해 **소리내어 읽어본다**:

```
[S] 그런데 [C]. 그래서 지금 [R1]이 벌어지고, 우리가 원하는 건 [R2]이다.
그렇다면 [Q]? 답은 [A].
```

이 낭독이 **자연스러우면 pass**. 한 군데라도 "어 이게 왜 여기 나오지?"가 생기면 warn, 문장 간 논리 점프가 있으면 fail.

이야기가 끊기는 흔한 원인:
- S에 논쟁적 주장이 들어감 (공통 지반 아님)
- C가 S와 무관한 별도 사실을 나열함
- R1/R2의 층위 불일치 (품질 vs 커리어 등)
- Q가 R1/R2 갭이 아니라 화자가 하고 싶은 질문에서 역산됨
- A가 Q의 직답이 아니라 옆으로 샘

판정 기록:

```markdown
| 체크 | 판정 | 메모 |
|------|------|------|
| S 공통지반 | pass/warn/fail | ... |
| C 진짜 변화 | pass/warn/fail | ... |
| R1/R2 갭 | pass/warn/fail | ... |
| Q 자연 생성 | pass/warn/fail | ... |
| A 직답 | pass/warn/fail | ... |
| A↔P 지지 | pass/warn/fail | ... |
```

**P 그루핑**: P가 5개 이상의 D를 가지면 서브P로 쪼갠다. P2 아래 P2-1, P2-2처럼. 서브P도 같은 규칙 적용 (순서 타입, 빌드업 검증).

P 레벨 배치 시 **순서를 먼저 결정**한다:
- 순서 타입: 시간순(인과) / 구조순(MECE 분해) / 정도순(중요도)
- 순서 근거: "P1→P2→P3인 이유: [청중 사고의 흐름 등]"
- 순서가 안 잡히면 그루핑이 잘못된 것.

##### Phase 1.5: 그루핑 게이트 — 자식 6개 이상이면 멈추고 범주화

**트리거**: 어느 노드든 직계 자식이 **7개 이상**이면 Phase 2로 넘어가지 않고 먼저 그루핑한다. L2 챕터 표 행 ≥ 7은 훅(`guardMintoNormalForm.mjs`)이 fail로 블록하므로 강제. 도입부의 C1~C7+, 결론 후보 등 **모든 레벨**에 동일 임계 적용.

왜: 민토 2번째 규칙("같은 그룹은 같은 종류다")은 자식이 5개를 넘으면 유지 불가능해진다. 7개를 나열하면 독자는 머릿속에서 무의식적으로 2~3개 묶음으로 재그룹핑한다. 그 재그룹핑을 화자가 대신 해주는 것이 이 게이트.

###### 그루핑 알고리즘

```
1. 자식 노드 전부 나열 (원문 그대로)
2. MECE 축 후보 뽑기 — 자식들이 공유하는 차원 찾기
   - 시간축 (원인/현상/결과)
   - 구조축 (전제/도구/운용/브릿지)
   - 주체축 (나/LLM/환경)
   - 층위축 (개념/방법/사례)
   - 강도축 (부탁/권장/강제)
3. 후보 2~3개를 사용자에게 제시
4. 선택된 축으로 2~4개 그룹으로 묶기
5. 각 그룹에 이름 = 결론 문장(카테고리 라벨 금지)
   - ❌ "문제점들", "방법들", "기타"  ← 카테고리 라벨
   - ✅ "바이브 코딩만으론 왜 안 되는지", "4가지 도구가 강제력 순으로 배열된다" ← 결론
6. 원본 노드는 그룹 아래 하위로 이동 (삭제 금지)
```

###### 그루핑 금지선

- **"기타" 그룹 금지** — "기타"가 생기면 축이 틀린 것. 다시 뽑는다.
- **한 그룹에 1개만 있는 경우 금지** — 한 그룹 안엔 최소 2개. 혼자면 다른 그룹에 합치거나 축을 바꾼다.
- **카테고리 라벨 금지** — 민토 1번째 규칙 위반. 그룹 이름은 반드시 "무엇이 어떻다"는 주장이어야 한다.
- **자식 수 균형** — 한 그룹만 5개고 다른 그룹이 1~2개면 축이 나쁨. 2~4개씩 비슷하게.

###### MECE 검증 (필수)

그루핑 제안 후 Phase 2 재진입 전에 **반드시** 표로 검증한다. warn 하나라도 있으면 축을 다시 뽑거나 경계 D를 재배치한다.

```markdown
| 체크 | 판정 | 근거 |
|------|------|------|
| **ME (Mutually Exclusive)** — 두 그룹에 동시에 속할 수 있는 D가 있는가? | pass/warn/fail | 경계 D 목록 |
| **CE (Collectively Exhaustive)** — 모든 D가 정확히 한 그룹에 속하는가? 누락된 D는? | pass/warn/fail | 미배치 D 목록 |
| **축 일관성** — 모든 그룹이 같은 축의 서로 다른 값인가? (시간축이면 전부 시간, 구조축이면 전부 구조) | pass/warn/fail | 축 섞임 있으면 fail |
| **그룹 수** — 2~4개인가? (5개 넘으면 상위 축 재탐색) | pass/warn/fail | |
| **그룹 크기** — 각 그룹 자식 수가 2~5개 범위인가? | pass/warn/fail | |
```

**ME 실패 유형**:
- 한 D가 두 그룹 설명 모두에 들어맞으면 → 그룹 결론이 겹치는 것. 그룹 결론을 더 좁히거나 D를 확정 배치.
- "원칙/사례 섞임"은 축이 층위축이 아닌 한 ME 위반 아님.

**CE 실패 유형**:
- 엔딩/훅/복선 등 수사 장치는 "결론 바깥"으로 빼서 CE 면제. 단, 분류 명시 필수.
- 누락 D가 발견되면 새 그룹을 만들지 말고 기존 그룹 결론을 확장할 수 있는지 먼저 확인.

###### 그루핑 후 순서·빌드업 재적용 (필수)

그루핑으로 새 (서브)P가 만들어지면 **모든 레벨에서 순서 타입과 근거를 다시 적용**한다. 그루핑은 "어떻게 묶을까"일 뿐 "어떤 순서로 읽힐까"는 별개 문제. 생략하면 MECE는 통과하지만 P들이 덜컥덜컥 끊긴다.

**필수 작업 — 각 신규 그룹마다 2개씩**:
1. **그룹 간 순서** — 형제 그룹들(G1, G2, G3...) 사이의 순서 타입(시간/구조/정도)과 근거 한 문장
2. **그룹 내 순서** — 그룹 안 D들 사이의 순서 타입과 근거 한 문장

**파일 표기**: 각 (서브)P 헤딩 바로 아래에

```markdown
**순서 타입 (G1 → G2 → G3)**: [시간/구조/정도]순 — [근거 1문장]
```

를 박는다. 이중 적용이므로 "그룹 간"과 "그룹 내" 두 곳 모두.

**빌드업 재검증**: 순서가 바뀌었으면 기존 →P 결론이 여전히 D 시퀀스에서 자연스럽게 귀결되는지 다시 본다. 순서 타입이 시간순이면 마지막 D가 결론에 가장 가까워야 하고, 정도순이면 가장 무거운 D가 끝에 와야 한다.

**근거 문장의 형태**: "X 순서로 놓아야 청중이 Y 반문에 빠지지 않는다" 또는 "X 순서여야 Y가 자연스럽게 착지한다" — 청중의 인지 흐름을 근거로 삼는다. "보기 좋아서" 같은 미적 이유는 금지.

###### 파일 포맷

그루핑 전후를 같이 보존한다(사용자가 되돌릴 수 있도록):

```markdown
### P2: [원본 결론]

<!-- Grouping: 축=구조축(전제/도구/운용/브릿지), 2026-04-12 -->

#### P2-1: [그룹1 결론 문장]
- (원본 D들 이동)

#### P2-2: [그룹2 결론 문장]
- ...
```

###### 대화 출력

그루핑 게이트가 발동하면 **이번 턴은 질문 대신 그루핑 제안**을 한다:

1. 자식 수와 현재 상태 한 줄 ("P2에 서브P 7개 — 그루핑 필요")
2. 축 후보 2~3개 (각 축으로 묶었을 때 어떻게 되는지 1줄씩)
3. "어느 축으로 가시겠어요? 아니면 제가 (a)로 진행할까요?"

사용자가 축을 골라주면 다음 턴에 실행, 그루핑 완료 후에 Phase 2 재진입.

##### Phase 2: P별 D 채우기 — 앞에서부터 하나씩

P1부터 순서대로 D를 수집하고 표로 만든다. **한 P가 완료되어야 다음 P로 간다.**

각 P에 대해:
1. D를 원문 그대로 수집한다
2. D의 순서 타입을 결정한다 (시간/구조/정도 + 근거)
3. D 빌드업이 P 결론에 도달하는지 검증한다
4. 도달하면 → 다음 P로
5. 불일치하면 → 갭 질문

##### 표 형식

**P마다 독립된 표 1개.** 여러 P를 하나의 큰 표에 몰아넣지 않는다. 표 앞에 P 헤딩(`### P1: [결론]`)을 두고 그 아래에 해당 P의 D만 나열한다. 가독성과 수정 편의 양쪽 이유 — 긴 표는 어느 P에 속하는 D인지 시선이 헤맨다.

```markdown
### P1: [P1 결론 문장]

| 접속사 | 낭독 | 원문 | 구체화 | 순서 | 위치 근거 | 요소 |
|--------|------|------|--------|------|----------|------|
| — | — | [원문 그대로] | 원칙 | 인과: 원인 | ... | D1 |
| 그래서 | [...마지막 한 절] 그래서 [...이 행 첫 한 절] | [원문 그대로] | 수치 | 인과: 조건 | ... | D2 |
| 그런데 | [...마지막 한 절] 그런데 [...이 행 첫 한 절] | [원문 그대로] | 사례 | 인과: 현상 | ... | D3 |
| — | — | ✓ 또는 ⚠ [P1 결론 문장] | | D 빌드업 도달 여부 | | →P1 |

### P2: [P2 결론 문장]

서브P가 있으면 그 아래 다시 작은 표로 분리.

#### P2-1: [서브P 결론]

| 접속사 | 낭독 | 원문 | 구체화 | 순서 | 위치 근거 | 요소 |
|--------|------|------|--------|------|----------|------|
| — | — | [원문] | 인용 | 구조: 첫째 | ... | D1 |
| — | — | ✓ [서브P 결론] | | | | →P2-1 |

#### P2-2: [서브P 결론]

| 접속사 | 낭독 | 원문 | 구체화 | 순서 | 위치 근거 | 요소 |
|--------|------|------|--------|------|----------|------|
| — | — | [원문] | 사례 | 구조: 둘째 | ... | D1 |
| — | — | ✓ [서브P 결론] | | | | →P2-2 |

#### →P2 종합: ✓ 또는 ⚠ [P2 결론 문장]

서브P들이 P2 결론에 도달했는지 판정 한 줄.
```

**금지**: 한 표 안에 `**P1**`, `**P2**`, `**P3**` 헤더 행을 섞어 넣지 않는다. 표는 P 하나당 하나.

- **구체화**: 원칙(추상적 주장) / 사례(구체적 예시) / 수치(정량 데이터) / 인용(논문·출처 명시) — 같은 P 아래에 원칙만 있으면 미완성, 사례·수치가 섞여야 완성
- **근거**: 논리(연역·인과 추론) / 경험(화자의 실무 체험) / 데이터(통계·벤치마크) / 논문(학술 인용) — 근거 유형이 한 가지만이면 편향 가능성
- **순서**: 이 D가 전개에서 맡는 역할 (인과: 원인, 인과: 현상, 구조: 첫째 요소 등)
- **→P**: D 빌드업이 P 결론에 도달하는지. P 결론 문장을 다시 적는다.

##### Phase 2.5: 결론 후보 발췌 — D 수집 중 엔딩 문장을 따로 모은다

D를 수집하다 보면 **근거이면서 동시에 결론(A' 또는 마무리 훅)으로 쓸 만한 문장**이 섞여 있다. 이것을 놓치면 결론이 비어버린다.

각 D를 추가할 때마다 두 번째 질문을 던진다:
1. "이 문장은 P의 근거로 쓰이는가?" → 표의 D
2. "이 문장은 독자가 가장 기억하길 바라는 한 방으로도 쓰일 수 있는가?" → **결론 후보**로 동시 발췌

두 조건에 모두 해당하면 양쪽에 기록한다 (D와 결론 후보 섹션).

###### 결론 후보 판정 기준

- **요약 문장**: P1~Pn을 한 문장으로 압축하는 주장
- **전환 문장**: 강의 전체의 의미가 한 번 뒤집히거나 확장되는 문장
- **기억 훅**: 청중이 강의장을 나가며 되뇌일 법한 인상 문장
- **대조/반전**: "X가 아니라 Y다" 형태의 단정
- **정의**: "Y 한다는 것은 Z라는 것이다" 형태의 핵심 정의

###### 파일 포맷

표 아래에 "결론 후보" 섹션을 둔다:

```markdown
## 결론 후보

| 출처 | 원문 | 역할 |
|------|------|------|
| P2-4 D8 | 그래서 좋은 하네스 엔지니어링을 할 줄 알게 된다는 것이 파트너 수준으로 갈 수 있다는 것입니다 | 정의 (핵심 브릿지) |
| P3-2 D15 | 가끔씩 자문합시다. "이걸 꼭 해야 하나?" | 기억 훅 |
```

Phase 3 진입 전, 이 후보들을 읽고 **A'(결론)을 조립**한다. A'는 도입부 A를 전개를 거친 뒤의 풍부해진 버전으로 재작성하되, 결론 후보들 중 **가장 강한 1~3개**를 뼈대로 삼는다.

##### Phase 3: 미배치 노드 재수용 — 빠뜨린 말을 꽂는다

Phase 2 완료 후 "미배치" 섹션에 남은 노드가 있으면, **미배치 = 말 못하는 게 아니라 사용자가 하고 싶었던 말이니까 자리를 찾아주는 것**이 원칙이다. 삭제하지 않는다.

각 미배치 노드에 대해:
1. **수사적 역할 판정** — 근거(D)인가, 비유/복선/훅(수사 장치)인가
2. **근거라면** → 가장 가까운 P의 표에 D로 추가
3. **수사 장치라면** → 해당 P에 "수사" 열로 부착. 역할 명시 (오프닝 훅 / 복선 / 비유 / 마무리 훅 등)
4. **어디에도 안 붙으면** → 그 노드가 실제로 하려던 말이 아직 피라미드에 없다는 신호. 새 P나 새 D로 승격 검토

미배치 섹션이 비어야 Phase 3 완료.

#### 3. 질문

##### 질문 생성 알고리즘

```
① 노드 간 모순 — 피라미드 안에서 충돌하는 쌍을 찾는다
   → LLM이 왜 충돌인지 논리를 밝힌다

② warrant 메움 — 점프하는 곳에 LLM이 논리를 채운다
   → 채운 논리의 전제를 역추적한다
   → 전제가 피라미드에 없으면 갭

③ MECE 빠진 차원 — 있어야 하는데 없는 것
```

우선순위: ① > ② > ③

##### 질문 포맷

```
제가 [X]라고 이해했는데 맞나요?
그렇다면 청중이 "[반문]"이라고 하면 뭐라고 대답할 건가요?
```

- LLM이 이미 알고 있는 답(사실, 근거)이 있으면 함께 적는다.
- 사용자의 논리에 대한 반문이어야 한다. 분류/배치 확인이 아니다.

##### 질문 제약

- Yes/No 금지. 열린 질문만.
- 사실 확인 금지. LLM이 답할 수 있으면 답하고 넘어간다.
- **트리 배치/구조 확인 금지 (직무유기)**. 다음 형태는 모두 금지:
  - "이거 어디 붙일까요?"
  - "α/β/γ 중 어느 구조로?"
  - "P3-1과 P3-2 순서를 바꿀까요?"
  - "D0이 서문이 맞나요, P3-2a 안이 맞나요?"
  - "이 그룹과 저 그룹 사이가 너무 멀지 않나요?"
  - **트리 토폴로지(노드 위치·형제 순서·부모 자식)에 대한 모든 질문**은 LLM이 판단해서 파일에 반영해야 한다. 사용자에게 묻는 것은 판단 회피.
- 한 문장으로 넘길 수 있으면 질문이 아니다.

##### 진짜 갭 질문 vs 직무유기 질문

| 직무유기 (금지) | 진짜 갭 (필수) |
|---|---|
| "D0을 서문에 둘까요 P3-2a에 둘까요?" (배치) | "D0이 '메타버스가 아니다'라고 단정하는데, 청중이 '메타버스도 초반엔 다 그렇게 말했어요'라고 하면 뭐라 답하실 건가요?" (내용 반문) |
| "P3-1c와 D0 사이 거리가 너무 멀지 않나요?" (구조) | "P3-1c의 '효율화 역할'을 청중이 'LLM이 더 나아지면 자동화될 텐데?'라고 반박하면 뭐라 답하실 건가요?" (논리 반문) |
| "두 P 순서를 바꿀까요?" (배치) | "D2의 '박보검' 비유를 청중이 '그건 너무 추상적이에요. 코드에서는 어떤 사례인가요?'라고 하면 뭐라 답하실 건가요?" (구체 요구) |

**진짜 갭 질문의 형태**:
> "청중이 [X라는 구체적 반문]을 던지면 뭐라 답하실 건가요?"
또는
> "[D번호]의 근거가 [경험/논리]뿐인데, 같은 결론에 [수치/논문/사례]를 더하려면 뭐가 있을까요?"

피라미드의 **내용**(주장·근거·반박)을 향한 질문만이 진짜 갭이다. 트리 자체에 대한 메타 질문은 모두 LLM이 해결할 일.

### 공통 규칙

- **제1원칙: 원문을 최대한 유지한다** — 표, 도입부, P, D 모두 사용자의 원문 그대로. 요약/정제/재작성 금지. 단어를 바꾸지 않는다. LLM이 "더 깔끔하게" 다듬는 순간 뉘앙스가 박살난다.
- **출처·수치는 LLM이 선제적으로 박는다** — 사용자가 "80%가 넘어요" 같은 거친 수치나 "벤치마크 추이" 같은 막연한 근거를 던지면, LLM이 해당 출처(Stack Overflow Developer Survey, JetBrains DevEcosystem, SWE-bench Verified 리더보드, GitHub Octoverse, arXiv 논문 ID 등)와 구체 수치를 알고 있는 한 D 작성 시점에 즉시 박는다. 사용자에게 "출처가 뭐예요?"로 다시 묻지 않는다. 사용자에게 묻는 건 LLM이 모르는 출처(예: 사내 통계, 강연자 본인 회사 사례)에만 한정. 알고 있는 통계 출처는 모두 LLM 책임.
- **파편 → 완전한 문장 정형화 (습니다체) — 강제 규칙** — 원문 셀은 **반드시 습니다체로 끝나는 완전한 문장**이어야 한다. 사용자가 파편·키워드·메모·반말로 던져도, LLM이 표에 박을 때는 무조건 습니다체 완전 문장으로 정형화한다. **검증 11**(훅)이 종결어미·길이·파편 패턴을 자동 검사한다.

  - **금지 형태** (모두 warn):
    - 키워드 나열 (`사람마다 다른 이유 — 예측 기계 × input × 경험`)
    - 제목/소제목 형태 (`4가지 도구 — 컨텍스트·스킬·에이전트·훅`)
    - 18자 미만 짧은 메모 (`스킬을 만드는 스킬`)
    - 종결어미 없는 토막 (`...해서 열받더라고`, `...3분 전의 나`)
    - 반말 (`...지`, `...거고`, `...해`, `...야`)
  - **허용 형태**:
    - `사람마다 LLM 결과가 다른 이유는 예측 기계 본질 + input 품질(경험) + 하네스 능력 세 변수가 곱해지기 때문입니다.`
    - `하네스에는 컨텍스트·스킬·에이전트·훅 네 가지 도구가 있고, 강제력 낮음에서 높음 순으로 쌓입니다.`
  - **변환 규칙**:
    - 반말 종결 → 습니다체 (`-어/-야/-지/-거고` → `-습니다/-입니다/-했습니다`)
    - 주어 보충 (없으면 문맥에서 추론하여 명시)
    - 명사·동사·감정 표현은 1:1 보존, 채우기 단어("사실", "그냥", "뭐", "이제")만 제거
    - 키워드 나열은 한 문장 안에 인과·관계로 풀어서 적기
  - **원칙**: 청중이 표를 그대로 읽어도 강연 한 단락으로 느껴져야 한다. 표가 메모장이 되면 minto가 아니라 아웃라이너가 된다.
  - **제거**: 쓸데없는 구어 채우기 단어("사실", "그냥", "뭐", "아", "이제", "그" 같은 필러)
  - **유지**: 원문의 명사·동사·감정 표현("열받더라고"→"열받았습니다", "복사 붙여넣기", "뭘 하러 가버리니까"→"뭘 하러 가버려서")
  - **변환 규칙**:
    - `-어/-야/-지/-거고` → `-습니다/-했습니다/-었습니다`
    - `그건` → `그것은` (지시어는 습니다체에 맞춰 조정)
    - 반말 종결 → 습니다체 종결
  - **절대 금지**:
    - 주어 삽입 금지 ("강연자가", "화자가" 등을 없던 자리에 넣지 않기)
    - 원문 명사·동사를 유의어로 교체 금지
    - 원문에 없던 설명/부연 추가 금지
  - **허용**: 문장 끊기(마침표), 지시 명확화 괄호 `(=fix 스킬)`
  - 원칙: **습니다체로 옷만 갈아입힌 복사 붙여넣기.** 독자가 "사용자가 직접 정제한 말"이라고 느껴야 한다.
- **구체화 강제** — D가 "원칙+논리"에만 머물면 미완성. 사용자에게 "사례 하나만", "수치 하나만", "출처 하나만"을 요구하는 질문을 만들어 끌어올린다. 구체화되지 못한 D는 그대로 두지 않는다.
- **나열이 나오면 항상 순서 결정** — 민토식 3가지(시간/구조/정도) 중 택1, 근거 명시 후 배치.
- **LLM이 배치 판단** — "어디 붙일까요?" 묻지 않음. 틀리면 사용자가 고침.
- **노드 삭제 금지** — 사용자 요청 전까지.
- **입력 규모에 맞게 밀도 조절** — 파편이 많고 방향 보이면 Phase 1을 한 턴에. 판단이 갈리는 지점에서만 멈춰 묻는다.
- **한 노드 아래 자식 2~5개** — 6개 이상이면 Phase 1.5 그루핑 게이트 발동. Phase 2 중에도 P 아래 D가 6개를 넘으면 그 자리에서 서브P로 쪼갠다.
- **P 완료 판정 후 자동 이동** — "다음 갈까요?" 묻지 않음.
- **메타 합의 질문 금지** — "다음 갈까요?", "이 방향 맞나요?", "진행할까요?" 모두 갭 질문 아님. 매 턴 마지막은 피라미드 내용을 향한 반문이어야 한다.

### 피라미드 3대 규칙 (레퍼런스)

1. **상위는 하위를 요약한다** — 카테고리 라벨 금지. 구체적 결론.
2. **같은 그룹은 같은 종류다** — 복수 명사 라벨 테스트.
3. **같은 그룹은 논리적 순서다** — 시간순 / 구조순 / 정도순.

### 파일 기반 운영

**항상 파일 기반.** 대화에는 발췌만.

#### 파일 생명주기

1. **첫 턴**: `docs/0-inbox/minto-{slug}.md`에 피라미드 파일 생성. slug는 주제에서 추출.
2. **매 턴 시작**: Read로 파일을 읽어 현재 피라미드 상태를 파악한다.
3. **매 턴 끝**: Edit/Write로 파일을 갱신한다. 파일이 SSOT.
4. **세션 종료**: 파일이 남아 있으므로 다음 세션에서 이어갈 수 있다.

#### 파일 포맷

```markdown
---
title: {주제}
stage: phase1 | phase2 | done
updated: {YYYY-MM-DD}
---

## 의도

이 글이 말하려는 건 [X]다.

## 도입부

- **S**: ...
- **C**: ...
- **R1**: ...
- **R2**: ...
- **Q**: ...
- **A**: ...

## 전개

| 접속사 | 낭독 | 원문 | 구체화 | 순서 | 위치 근거 | 요소 |
|--------|------|------|--------|------|----------|------|
| — | — | [결론] | — | — | — | **P1** |
| — | — | [원문] | ... | ... | ... | D1 |
| — | — | ✓/⚠ [결론] | | | | →P1 |
...

## 결론

- **A'**: ...

## 열린 질문

- [ ] ...
```

#### 대화 출력 규칙

매 턴 대화에는 **전체 피라미드를 복사하지 않는다.** 다음을 순서대로 출력:

1. **상하위 연결 표 (최대 3개 항목)** — 초점이 되는 노드를 중심으로 **A → 상위 P → 해당 노드 → (있으면) 하위 D**의 수직 경로를 표로 보여준다. 수평 형제는 숨긴다.
2. **질문** — 발췌한 경로의 빈틈을 향한 반문.
3. 파일 경로 한 줄 (처음 생성 시, 또는 사용자가 물을 때)

##### 상하위 연결 규칙

- **항상 A부터 시작** — 루트가 없으면 독자가 "어디 이야기인지" 모른다.
- **수직 연결 필수** — A → P → (서브P) → D 경로가 보여야 한다. 중간을 건너뛰지 않는다.
- **수평 형제는 숨긴다** — P1 이야기를 할 때 P2, P3는 안 보여준다. 질문 대상 경로만.
- **한 턴에 최대 3개 항목**만 출력. 항목 = 독립된 연결 경로. 3개 넘으면 "가장 급한 3개만" 고르고 나머지는 "그 외 N개 (파일 참조)"로 요약.
- **신규/변경 항목 우선** — 이번 턴에 추가·변경된 것을 먼저.
- **약한 경로 우선** — →P가 ⚠ / 구체화·근거 편향인 것.

##### 예시

✅ 좋은 출력 (1개 항목, 수직 연결):

```
| 요소 | 원문 | 구체화 | 근거 |
|------|------|--------|------|
| A | 하네스를 배우고 경험을 쌓아라 | | |
| └ P3 | 남는 것을 알고 내일부터 시작한다 | | |
|   └ P3-1 | 남는 건 문제의 이해도와 구조화 능력 | | |
|     └ D6 | Claude Code Hackathon Top5 중 3명 비개발자 | 인용 | 데이터 |
```

❌ 나쁜 출력: 전체 피라미드 복사, 수평 형제 나열, 12개 평행 표.

##### "가장 약한" 판정 기준 (우선순위)

⚠ > 구체화 편향(원칙만) > 근거 편향(한 종류만)
