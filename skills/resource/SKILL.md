## /resource — 외부 조사 기반 참고 문서 생성

> **목적**: 현재 맥락에서 필요한 주제를 **외부 검색으로 깊이 조사**하여 프로젝트 참고 자료를 만든다.
> **산출물**: `docs/3-resources/`에 저장되는 조사 문서.
> **핵심 가치**: 프로젝트에 없는 외부 지식을 끌어와서, 맥락에 맞게 정제하는 것.

---

### /explain과의 구분

| | /explain | /resource |
|---|---|---|
| **소스** | 내부 (코드, git, 프로젝트 문서) | **외부** (웹 검색, 스펙 문서, 논문, 블로그) |
| **질문** | "우리가 왜 이렇게 했지?" | "세상에서는 이걸 어떻게 하지?" |
| **저장** | `docs/0-inbox/[explain]` | `docs/3-resources/` |

내부 맥락 해설이 필요하면 /explain, 외부 지식 조사가 필요하면 /resource.

---

### Step 0: 주제 결정

1. **인자가 있으면** → 그 주제를 조사한다 (예: `/resource WAI-ARIA TreeGrid pattern`)
2. **인자가 없으면** → 현재 대화 맥락에서 조사가 필요한 주제를 추론한다:
   - 구현 중 막힌 기술적 질문이 있으면 → 그것
   - 설계 결정에 외부 근거가 필요하면 → 그것
   - 판단 불가 → 사용자에게 묻는다

주제가 결정되면 한 줄로 선언한다:
```
🔍 조사 주제: [주제명] — [왜 이 조사가 필요한지 1문장]
```

---

### Step 1: 검색 전략 수립

조사를 시작하기 전에 **검색 방향을 먼저 설계**한다. 무작정 검색하면 표면적 결과만 모이기 때문이다.

1. **핵심 질문 3-5개** 도출 — 이 주제에서 꼭 답해야 할 것들
2. **검색 키워드** 결정 — 영어 + 한국어, 일반 용어 + 전문 용어 모두
3. **신뢰할 소스 우선순위** 설정:

| 우선순위 | 소스 유형 | 예시 |
|---------|----------|------|
| 1 | 공식 스펙/표준 | W3C, ECMA, RFC |
| 2 | 공식 문서 | MDN, React docs, 라이브러리 README |
| 3 | 전문가 블로그/발표 | 핵심 메인테이너 블로그, 컨퍼런스 토크 |
| 4 | 커뮤니티 논의 | GitHub issues, Stack Overflow, RFC discussions |
| 5 | 일반 블로그/튜토리얼 | 개념 설명, 사용 사례 |

검색 전략을 사용자에게 보여주고 확인받는다. 방향이 맞으면 검색을 시작한다.

---

### Step 2: 외부 검색 + 수집

**WebSearch와 WebFetch를 적극적으로 사용한다.** 이 스킬의 핵심은 외부 검색이다.

#### 검색 규칙

- **최소 3회 이상** 검색한다 — 첫 검색 결과만으로는 깊이가 부족하다
- **꼬리를 물고 파고든다** — 검색 결과에서 나온 용어/링크를 다시 검색하여 깊이를 확보한다
- **다른 각도로 검색한다** — 같은 주제를 다른 키워드/관점으로 재검색
- **원문을 읽는다** — 검색 스니펫이 아닌, WebFetch로 원문 페이지를 읽어 맥락을 확보한다
- **출처를 기록한다** — 모든 정보에 출처 URL을 남긴다

#### 수집하면서 메모할 것

- 핵심 개념과 정의
- 다른 접근 방식들과 각각의 trade-off
- 프로젝트에 적용 가능한 패턴
- **흥미로운 점** — 예상과 다른 사실, 반직관적 인사이트, 숨겨진 함정
- 논쟁이 있는 부분과 각 입장

---

### Step 3: 문서 구조 설계

**Why → How → What → If** 골든서클 구조를 따른다.

| 섹션 | 질문 | 이 스킬에서의 의미 |
|------|------|-------------------|
| **Why** | 왜 이 주제가 중요한가? | 배경, 이 기술/패턴이 등장한 이유, 해결하려는 문제 |
| **How** | 어떻게 작동하는가? | 메커니즘, 구조, 핵심 원리 |
| **What** | 실제로 무엇이 있는가? | 구체적 구현, API, 패턴, 사례 |
| **If** | 우리 프로젝트에 어떤 의미인가? | 적용 방향, 제약, 대안, 주의점 |

각 섹션에 **Mermaid 다이어그램이 들어갈 수 있는 곳을 식별**한다.

---

### Step 4: 문서 작성

#### 문서 구조

```markdown
# [주제명] — [한 줄 설명]

> 작성일: YYYY-MM-DD
> 맥락: [이 조사가 필요해진 배경 1문장]

> **Situation** — [모두가 동의하는 현재 상태. 사실 기반, 1-2문장]
> **Complication** — [그 상태에서 발생한 문제·변화·긴장. "그런데…"]
> **Question** — [이 문서가 답하려는 핵심 질문. 자연스럽게 본문으로 연결]
> **Answer** — [한 줄 결론. 본문의 요약이자 독자가 기대할 내용]

---

> 이 주제를 프로젝트에서 직접 확인하려면?

1. [진입점 — 관련 파일 경로, 설정, 또는 실행 커맨드]
2. [첫 번째 조작 — 무엇을 하면 무엇이 보이는가]
3. [핵심 시나리오 — 조사 내용의 효과를 체감할 수 있는 최소 동선]
4. [확인 포인트 — 이것이 제대로 적용되면 어떤 상태인가]
```

#### 톤과 깊이

- **독자**: 이 주제를 처음 접하지만 기술적 소양이 있는 개발자
- **톤**: 기술 문서. 감탄사/수사적 질문 금지. 사실과 근거만.
- **깊이**: 표면적 개요가 아닌, "왜 이런 선택을 했는지"까지 설명할 수 있어야 한다
- **프로젝트 연결**: If 섹션에서 반드시 현재 프로젝트 맥락과 연결한다
- **Mermaid**: 최소 2개. explain 스킬의 다이어그램 규칙을 따른다

---

### Step 5: 저장 + 보고

1. **디렉토리 확인**: `docs/3-resources/` 없으면 생성
2. **저장**: `docs/3-resources/{주제명}.md`
   - 파일명은 주제를 명확히 나타내는 kebab이 아닌 camelCase 또는 영문 명사
   - 예: `docs/3-resources/ariaTreeGridPattern.md`
3. **보고**: 사용자에게:
   - 핵심 발견 3-5줄 요약
   - 가장 흥미로운 인사이트 1개 하이라이트
   - 파일 경로
