## /conflict — 대립해소도 (Evaporating Cloud)

> **이론적 기반**: Eliyahu Goldratt, TOC Thinking Processes — Evaporating Cloud
> 양립 불가능해 보이는 두 방향 사이에서, "둘 중 하나를 포기"하는 대신 **숨겨진 전제를 깨서 양립 가능하게** 만든다.

### 핵심 원리

대립은 보통 "A냐 B냐"로 보이지만, 실제로는 **전제가 만든 착시**인 경우가 많다. Cloud의 각 화살표에는 "왜 이게 필수인가?"라는 전제가 숨어 있고, 그 전제 하나만 깨지면 충돌이 증발(evaporate)한다.

```
              [A: 공통 목표]
             /              \
        ①               ②
           /                  \
    [B: 필요조건]        [C: 필요조건]
           \                  /
        ③               ④
           \                /
     [D: 행동] ←⑤ 충돌→ [D': 행동]
```

화살표 ①~⑤ 각각에 전제가 있다. 전제를 깨면 화살표가 끊어지고, 충돌이 사라진다.

## 진입 경로

| 경로 | 트리거 | 맥락 |
|------|--------|------|
| **독립 호출** | `/conflict` 직접 입력, "A vs B 딜레마" | 전체 5단계 진행 |
| **`/discuss` 드릴다운** | 제약·전제가 2턴 연속 🟡 정체 | discussion 맥락에서 Cloud 구성 후 복귀 |

## Step 0: 진입

1. `/discuss`에서 라우팅된 경우: 대화 맥락에서 충돌 지점을 파악한다.
2. 독립 호출인 경우: 사용자가 제시한 딜레마를 파악한다.

## Step 1: 공통 목표 (A) 정의

양쪽 모두 진짜 원하는 것을 찾는다. 충돌하는 두 방향이 **공유하는 상위 목적**이다.

```
"[A]를 달성하고 싶다."
```

양쪽이 동의하지 않는 목표는 공통 목표가 아니다. 충분히 올라가면 반드시 합류점이 있다.

## Step 2: 필요조건 (B, C) 도출

공통 목표를 위해 각각 필요한 것을 명시한다.

```
"A를 위해 B가 필요하다." (한쪽의 가치/방향)
"A를 위해 C가 필요하다." (다른쪽의 가치/방향)
```

B와 C 자체는 충돌하지 않는다. 둘 다 A를 위해 필요한 것이다.

## Step 3: 행동 (D, D') 명시 + 충돌 확인

각 필요조건을 충족하기 위한 구체적 행동을 적고, 양립 불가능함을 확인한다.

```
"B를 충족하려면 D를 해야 한다."
"C를 충족하려면 D'를 해야 한다."
"D와 D'는 양립할 수 없다." ← 이것이 충돌
```

여기까지가 Cloud 다이어그램의 완성이다. 사용자에게 다이어그램을 보여주고 확인을 받는다.

**확인 형식:**
```
              [A: ...]
             /         \
      [B: ...]       [C: ...]
             \         /
       [D: ...] ↔충돌↔ [D': ...]

이 구조가 맞나요?
```

## Step 4: 전제 나열

Cloud의 5개 화살표 각각에 "왜 이 연결이 필수인가?"의 숨겨진 전제를 나열한다. 이것이 EC의 핵심이다.

| 화살표 | 연결 | 전제 (왜 필수인가?) |
|--------|------|---------------------|
| ① | A → B | "A를 위해 B가 필수인 이유: ..." |
| ② | A → C | "A를 위해 C가 필수인 이유: ..." |
| ③ | B → D | "B를 충족하려면 D밖에 없는 이유: ..." |
| ④ | C → D' | "C를 충족하려면 D'밖에 없는 이유: ..." |
| ⑤ | D ↔ D' | "D와 D'가 양립 불가능한 이유: ..." |

각 화살표에 전제가 여러 개일 수 있다. 전부 적는다.

**전제를 잘 찾는 팁:**
- "정말 이것밖에 방법이 없나?" (③, ④에 효과적)
- "항상 그런가, 특정 조건에서만 그런가?" (⑤에 효과적)
- "이게 사실이라는 근거가 있나, 그냥 당연하다고 느끼는 건가?" (①, ②에 효과적)

## Step 5: 전제 깨기 → 돌파구 (Injection)

나열된 전제를 하나씩 검토한다. "이 전제가 거짓이라면?" 또는 "이 전제가 성립하지 않는 조건이 있다면?"

```
전제 ③-1: "B를 충족하려면 D밖에 없다"
→ 깨기: "D가 아닌 D''로도 B를 충족할 수 있다면?"
→ Injection: [구체적 대안]
→ 검증: D''가 C와 양립 가능한가? ✅
```

깨기 쉬운 순서: ③④ (행동의 유일성) > ⑤ (양립 불가) > ①② (필요성 자체)

③④가 가장 깨기 쉬운 이유: "이 방법밖에 없다"는 전제는 대부분 시야의 한계에서 온다. 다른 방법을 찾으면 충돌 자체가 사라진다.

### Injection 검증

돌파구가 나오면 3가지를 확인한다:
1. **B를 충족하는가?** — 한쪽 가치를 훼손하지 않는가
2. **C를 충족하는가?** — 다른쪽 가치를 훼손하지 않는가
3. **새로운 충돌을 만들지 않는가?** — 부작용 체크

3가지 모두 통과하면 돌파구가 확정된다.

## 산출물

### `/discuss`에서 라우팅된 경우

Cloud 다이어그램 + 깨진 전제 + Injection을 인라인으로 제시하고 `/discuss`으로 복귀한다.
- 깨진 전제 → discussion의 **전제** 요소 갱신
- Injection → discussion의 **목표** 또는 **해결** 요소에 반영

### 독립 호출의 경우

결과를 대화에서 직접 제시한다:

```markdown
## 대립해소도: [제목]

### Cloud
[다이어그램]

### 전제 분석
[5개 화살표별 전제 목록]

### Injection (돌파구)
- 깨진 전제: [어떤 전제가 거짓이었는가]
- 돌파구: [구체적 대안]
- 검증: B 충족 ✅ / C 충족 ✅ / 부작용 없음 ✅
```
