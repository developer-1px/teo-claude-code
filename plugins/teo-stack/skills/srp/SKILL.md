## 목적

파일명이 말하는 목적과 실제 코드의 책임을 비교하여, 불일치하는 책임을 분리한다.

일반적인 "함수가 길다", "중복 코드" 같은 코드 스멜은 다루지 않는다. 이 스킬은 **파일 단위의 책임 경계**만 다룬다.

## 동작: 분석 → 리팩토링

### Phase 1: 분석 (빈칸 채우기)

아래 템플릿을 순서대로 채운다. 순서가 중요하다 — 확실한 사실을 먼저 쓰고, 추론을 거쳐, 결론은 마지막에 도출한다. 결론을 먼저 정하고 근거를 끼워맞추는 사고를 방지하기 위한 구조다.

```
## 단일 책임 점검

파일: ____
파일명이 말하는 목적: ____

### 사실 수집 (코드에서 확인)

이 파일이 하는 일:
1. ____
2. ____
...

### 추론 (사실 → 판단)

| Line | 하는 일 | 목적과의 관계 | 판정 | 확신 | → 귀속 |
|------|---------|-------------|------|------|--------|
| ____ | ____ | ____ | ✅/❌/⚠️ | C/K/X | ____ |

### 결론

실행:
| 책임 | → 귀속 파일명 |
|------|-------------|
| ____ | ____ |

보류 (@FIXME):
| 책임 | 애매한 이유 | 판단 조건 |
|------|-----------|----------|
| ____ | ____ | ____ |
```

#### 채우는 순서

1. **파일명 → 목적**: 파일명만 보고 이 파일이 해야 할 일을 한 문장으로 쓴다
2. **사실 수집**: 코드를 읽고 실제로 하는 일을 나열한다 (line range 포함)
3. **추론 표**: 각 책임마다 근거(목적과의 관계)를 먼저 쓰고, 판정과 확신도를 그 다음에 적는다
4. **결론**: 확신도에 따라 실행/보류로 분류한다

#### 확신도 (Cynefin)

판정의 확실함을 3단계로 표기한다. 확신도가 행동을 결정한다 — 애매하면 안 한다.

| 확신 | Cynefin | 의미 | 행동 |
|------|---------|------|------|
| **C** | Clear | 파일명 목적과 명백히 일치/불일치 | 즉시 실행 |
| **K** | Complicated | 분석하면 답이 나오지만 판단이 필요 | AI 판단 제시 + 사용자 확인 후 실행 |
| **X** | Complex | 옮겨봐야 아는 것 | 실행하지 않음. `@FIXME(srp)` 주석만 남김 |

#### @FIXME 주석 형식

Complex(X) 판정을 받은 책임은 코드에 주석으로 남긴다:

```ts
// @FIXME(srp): [애매한 이유] — 판단 조건: [언제 Clear가 되는가]
```

분석 결과를 사용자에게 보여주고 확인을 받는다.

### Phase 2: 리팩토링 실행

사용자가 분석에 동의하면, Clear(C)와 확인된 Complicated(K) 항목만 리팩토링을 수행한다.

#### 실행 원칙

- **원자적 이동**: 관련 코드를 한 번에 옮긴다. 중간 상태에서 빌드가 깨지지 않도록 한다
- **인터페이스 최소화**: 분리된 파일이 원본에 노출하는 API는 최소한으로 한다
- **파일명 = export**: 새 파일명은 주 export 식별자와 일치해야 한다
- **import 경로 갱신**: 분리 후 기존 import를 모두 갱신한다
- **@FIXME 삽입**: Complex(X) 항목은 해당 코드 위에 주석을 남긴다

#### 실행 순서

1. 새 파일 생성 (분리 대상 코드 이동)
2. 원본 파일에서 제거 + 새 파일 import
3. Complex(X) 항목에 @FIXME 주석 삽입
4. 다른 파일의 import 경로 갱신
5. 프로젝트의 정적 검증(타입체크/린트/빌드) 실행하여 회귀 확인

#### 검증

리팩토링 완료 후 프로젝트의 정적 검증(예: `tsc --noEmit`, `pnpm typecheck`, `npm run build`, `eslint` 중 있는 것)을 실행한다. 실패하면 수정한다.
