# Reframe: CMS 드래그앤드롭 노드 이동

## 지금까지의 흐름

1. **dnd 플러그인 수정** — 결과: 드래그는 되지만 드롭 후 노드 위치가 반영 안 됨
2. **store command 추가** — 결과: 데이터는 이동되지만 화면에 반영이 안 되거나 순서가 틀림
3. **렌더링 순서 수정** — 여기서 멈춤

패턴: **수정이 수정을 낳고 있다.** 플러그인 → store → 렌더링으로 점점 하위 레이어를 건드리며 범위가 확산 중. 원래 의도("노드를 드래그로 옮기기")에서 멀어져 시스템 내부를 뜯고 있다.

---

## 재점검

| 요소 | 재점검 결과 | 기존 가정과 다른 점 |
|------|-------------|---------------------|
| **목적** | CMS에서 노드 순서/위치를 드래그로 변경하고 싶다 | 없음 — 목적 자체는 명확 |
| **이상적 결과** | 드래그 → 드롭 → store 반영 → UI 즉시 갱신 | "렌더링 순서"까지 신경 쓸 일이 아니어야 한다 |
| **현실** | dnd 플러그인, move command, 렌더링 순서 세 곳을 동시에 수정 중 | **세 레이어를 동시에 건드린다는 것 자체가 설계 문제의 증상**. 정상이라면 move command 하나로 끝나야 한다 |
| **문제** | 진짜 갭: store의 노드 순서 모델과 렌더링이 쓰는 순서 소스가 일치하지 않는 것 | 기존 가정은 "dnd 플러그인이 부족하다"였지만, 플러그인은 이벤트만 발행하면 된다. 문제는 그 아래다 |
| **원인** | `NormalizedData`의 `rootIds` 또는 `children` 배열 순서가 렌더링의 SSOT가 아닐 가능성. 또는 `getVisibleNodes`가 별도 정렬을 하고 있어서 store 순서와 화면 순서가 분리되어 있을 수 있다 | "dnd가 안 된다"가 아니라 **"순서의 SSOT가 하나가 아니다"**가 근본 원인 |
| **목표** | 순서의 단일 소스를 확보하는 것. store의 children/rootIds 순서 = 렌더링 순서가 되면 dnd 플러그인은 move command만 발행하면 끝 | 고칠 곳이 세 군데가 아니라 **한 군데**(순서 SSOT)여야 한다 |

## 진단: 왜 헤맸는가

**범위 착각 + 증상-원인 혼동.** "드래그가 안 된다"는 증상을 dnd 플러그인에서 시작해 한 겹씩 벗기며 쫓아갔지만, 진짜 문제는 순서 데이터의 SSOT가 분리되어 있다는 구조 문제다. 증상이 나타난 곳(플러그인)과 원인이 있는 곳(store/engine의 순서 모델)이 다르다.

---

## 재정의된 문제

**store의 노드 순서와 렌더링 순서가 분리되어 있어서, 어떤 command를 실행해도 화면에 올바르게 반영되지 않는다.**

## 새 접근

1. `getVisibleNodes`(engine)가 순서를 어디서 가져오는지 확인한다. store의 `children` 순서를 그대로 쓰는지, 아니면 별도 정렬/필터를 하는지.
2. 만약 별도 정렬이 있다면, 그것을 제거하고 store 순서를 SSOT로 만든다.
3. 순서 SSOT가 확보되면, dnd 플러그인은 `move` command 하나만 발행하면 된다. store command 추가도, 렌더링 순서 수정도 필요 없다.

## 기존 접근과의 차이

기존: dnd 플러그인 → store command → 렌더링 순서를 **각각 따로 고침** (바깥에서 안으로 파고들기).
새 접근: **순서 SSOT를 먼저 하나로 통일**하고, 나머지는 자동으로 따라오게 함 (안에서 바깥으로).

이 방향이 맞는지 확인해 달라. 이미 한번 틀린 상태니까 — `getVisibleNodes`의 실제 구현을 같이 보면서 순서 소스가 정말 분리되어 있는지부터 확인하는 게 좋겠다.
