## DnD 노드 이동 — 범위 확산 진단

### 현상 요약

DnD로 노드를 이동하려는데 수정 범위가 계속 확산되고 있다:

1. dnd 플러그인 수정
2. store command 추가
3. 렌더링 순서 수정

세 레이어를 모두 건드리고 있다.

### 판단: 범위 확산이 정상인 경우와 아닌 경우

**정상일 수 있는 경우**: DnD 노드 이동은 본질적으로 여러 레이어를 관통하는 기능이다. 플러그인(인터랙션 감지) → store command(데이터 변경) → 렌더링(결과 반영)은 레이어 구조상 자연스러운 흐름이다. 각 레이어에서 "없는 것을 추가"하는 거라면 정상이다.

**비정상인 경우**: 한 레이어를 고쳤더니 다른 레이어가 깨져서 따라 고치는 패턴이라면 문제다. 이건 scope creep이 아니라 **설계 결함의 증상**이다.

### 자문할 질문 3가지

1. **각 레이어 수정이 독립적인가?** dnd 플러그인 수정과 store command 추가가 각각 독립된 이유로 필요한 건지, 아니면 플러그인 수정이 실패해서 store로 내려간 건지. 후자라면 플러그인 설계가 문제다.

2. **렌더링 순서를 왜 건드리는가?** store command로 데이터가 올바르게 바뀌면 렌더링은 자동으로 따라와야 한다. 렌더링 순서를 직접 건드린다는 건 store의 노드 순서(NormalizedData의 루트 ID 배열이나 children 배열)가 이동을 제대로 반영하지 못하고 있다는 뜻일 수 있다.

3. **"노드 이동"의 정의가 명확한가?** move = 부모의 children 배열에서 remove + 대상 위치에 insert. 이게 하나의 command로 원자적으로 처리되고 있는지, 아니면 여러 단계로 쪼개져서 중간 상태가 발생하는 건지.

### 권장 접근

**렌더링 순서 수정은 멈춰라.** 렌더링은 store 상태의 결과여야 한다. 렌더링을 직접 건드리는 건 증상 패칭이다.

대신 이 순서로 검증하라:

1. **Store command부터 확인**: `moveNode` command가 NormalizedData를 정확히 변환하는지 단위 테스트로 검증. children 배열의 순서가 의도대로 바뀌는지.
2. **Engine 순회 확인**: `getVisibleNodes`가 변경된 store 상태를 올바른 순서로 반환하는지.
3. **플러그인은 drop 시 command를 dispatch하는 역할만**: dnd 플러그인이 직접 순서를 조작하면 안 된다. drop 이벤트 → moveNode command dispatch → store 업데이트 → 리렌더.

이 구조에서 렌더링 순서를 별도로 건드릴 이유가 없다. 건드리고 있다면, store command가 순서를 제대로 반영하지 못하고 있는 것이다.

### 한 줄 결론

세 레이어를 건드리는 것 자체는 문제가 아니다. **렌더링 순서를 직접 수정하는 것**이 문제다 — store command가 데이터 순서를 올바르게 바꾸면 렌더링은 따라온다. command부터 다시 검증하라.
